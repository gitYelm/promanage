## 新发现的安全问题

### 🔴 高危：两步验证密钥泄露

**问题描述**:
`TwoFactorService.getUserTwoFactorStatus()` 方法返回了用户的 `twoFactorSecret` 字段，这是 TOTP 算法的密钥。

**影响范围**:
- `POST /auth/twoFactor/verify` - 验证两步验证码时
- 内部服务调用

**风险评估**:
- 🔴 **严重程度**: 高危
- 攻击者获取此密钥后可以生成有效的 2FA 验证码
- 完全绕过两步验证安全机制

**修复方案**:
将 `getUserTwoFactorStatus()` 拆分为两个方法：
1. `getUserTwoFactorStatus()` - 仅返回启用状态（供外部使用）
2. `getUserTwoFactorSecret()` - 返回密钥（仅供内部使用）

**修复代码**:
```typescript
// two-factor.service.ts

// ✅ 安全：仅返回状态
async getUserTwoFactorStatus(userId: string): Promise<{ enabled: boolean }> {
  const user = await this.prisma.sysUser.findUnique({
    where: { userId: BigInt(userId) },
    select: { twoFactorEnabled: true },
  })
  return {
    enabled: user?.twoFactorEnabled ?? false,
  }
}

// ✅ 内部方法：返回密钥
async getUserTwoFactorSecret(userId: string): Promise<string | null> {
  const user = await this.prisma.sysUser.findUnique({
    where: { userId: BigInt(userId) },
    select: { twoFactorSecret: true },
  })
  return user?.twoFactorSecret ?? null
}
```

**调用方修复**:
```typescript
// auth.service.ts
const secret = await this.twoFactorService.getUserTwoFactorSecret(tempData.userId)
```

**状态**: ✅ 已修复

### 🟡 中危：2FA 设置接口返回密钥

**接口**: `GET /auth/twoFactor/setup`

**问题描述**:
该接口返回新生成的 `secret` 字段，用于用户设置两步验证。

**返回数据**:
```json
{
  "globalEnabled": true,
  "userEnabled": false,
  "secret": "JBSWY3DPEHPK3PXP",  // ← TOTP 密钥
  "qrCode": "data:image/png;base64,..."
}
```

**风险分析**:
- ⚠️ 这是 **2FA 设置流程的必要步骤**
- 用户需要这个密钥来配置 Google Authenticator 等应用
- 密钥只在设置时返回一次，用户需要立即保存
- 接口需要 JWT 认证，只有登录用户才能访问

**业务合理性**:
- ✅ 符合 TOTP 标准流程（RFC 6238）
- ✅ 与 Google、GitHub、AWS 等大厂的 2FA 设置流程一致
- ✅ 密钥通过 HTTPS 加密传输
- ✅ 需要用户验证码确认后才真正启用

**安全建议**:
1. ✅ 已添加代码注释说明这是必要的业务流程
2. ⚠️ 建议前端在用户确认保存后立即清除内存中的 secret
3. ⚠️ 建议添加操作日志记录 2FA 设置行为
4. ⚠️ 建议限制该接口的调用频率（防止暴力枚举）

**修复代码**:
```typescript
// auth.controller.ts
return {
  globalEnabled: true,
  userEnabled: enabled,
  // 注意：secret 只在设置时返回一次，用户需要立即保存
  // 这是 2FA 设置流程的必要步骤
  secret,
  qrCode,
}
```

**状态**: ✅ 已添加注释说明，业务合理

### ✅ 系统配置管理（业务合理）

**接口**: 
- `GET /system/config` - 查询配置列表
- `GET /system/config/:configId` - 查询配置详情

**敏感配置**:
```
sys.mail.password          - 邮箱密码/授权码
sys.storage.oss.accessKey  - OSS 访问密钥
sys.storage.oss.secretKey  - OSS 访问密钥
sys.git.token              - GitHub Token
```

**当前实现**:
- ✅ 配置接口返回明文值（管理员需要查看和修改配置）
- ✅ 接口需要 `system:config:list` 权限（仅超级管理员可访问）
- ✅ 操作日志记录配置修改（已脱敏）
- ✅ 配置值存储在数据库中

**业务合理性分析**:

1. **这是管理员后台系统**
   - 管理员的职责就是配置系统参数
   - 如果看不到配置值，管理员无法验证配置是否正确
   - 脱敏后管理员无法修改配置（不知道原值）

2. **符合行业标准**
   - 宝塔面板、PHPMyAdmin、WordPress 后台都是明文显示
   - 阿里云、腾讯云控制台也是明文显示密钥
   - 几乎所有后台管理系统都采用这种方式

3. **权限控制充分**
   - 需要超级管理员权限才能访问
   - 已有完整的 RBAC 权限体系
   - JWT 认证 + 可选的两步验证

4. **脱敏无法提高安全性**
   - 如果管理员账号被盗，攻击者可以修改配置获取明文
   - 脱敏只是"掩耳盗铃"，反而降低了可用性
   - 真正的安全边界是防止未授权访问，而不是防止管理员查看

**真正的安全措施**:

✅ **已实现**:
- RBAC 权限控制（只有超级管理员可访问）
- JWT 认证
- 操作日志记录（配置修改已脱敏）
- 密码加密存储
- 两步验证支持

⚠️ **可以加强**:
- 强制管理员启用 2FA
- 管理员密码强度要求
- 管理员登录 IP 白名单
- 敏感配置修改告警
- 异常登录检测

**结论**: 
当前实现符合业务需求和行业标准，无需修改。真正应该关注的是管理员账号安全和访问控制，而不是对管理员隐藏配置。

**状态**: ✅ 业务合理，无需修改
