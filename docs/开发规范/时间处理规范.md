# 时间处理规范

## 概述

本项目采用 **UTC 存储 + 本地显示** 的时间处理策略，这是后台管理系统的标准做法。

## 时区基础概念

**UTC（协调世界时）** 是全球统一的时间标准，不受地区影响。

**GMT（格林威治标准时间）** 是以英国伦敦格林威治天文台为基准的时间。日常使用中 **GMT = UTC**，只是 UTC 基于原子钟更精确，是现代国际标准。

**时区** 是相对于 UTC 的偏移量：
- 北京时间 = UTC + 8 小时（东八区）
- 纽约时间 = UTC - 5 小时（西五区）
- 伦敦时间 = UTC + 0 小时（零时区）

**换算示例：**
```
UTC 00:00 = 北京 08:00 = 纽约 19:00(前一天)
UTC 08:00 = 北京 16:00 = 纽约 03:00
UTC 16:00 = 北京 00:00(次日) = 纽约 11:00
```

**为什么用 UTC 存储？**
- 全球统一，不受夏令时影响
- 方便跨时区系统协作
- 数据库、服务器、API 都用同一标准

## 数据流

```
PostgreSQL (UTC) → Prisma (Date) → API (ISO 8601) → 前端 (本地时间)
```

| 层级 | 格式 | 示例 |
|------|------|------|
| 数据库 | TIMESTAMP (UTC) | `2026-01-16 08:30:00` |
| Prisma | JS Date 对象 | `Date` |
| API 响应 | ISO 8601 字符串 | `2026-01-16T08:30:00.000Z` |
| 前端显示 | 本地时间 | `2026-01-16 16:30:00` |

## 为什么数据库和管理后台时间不一样？

这是**正常现象**，不是 Bug：

- **数据库/Prisma Studio**：显示 UTC 时间（世界标准时间）
- **管理后台**：显示北京时间（UTC + 8 小时）

例如北京时间 `16:30:00`，在数据库中存储为 UTC `08:30:00`，相差 8 小时。

## 各层处理方式

### 1. 数据库层 (PostgreSQL)

使用 `TIMESTAMP` 类型，不带时区：

```sql
create_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP
```

### 2. Prisma Schema

```prisma
model SysUser {
  createTime DateTime? @default(now()) @map("create_time") @db.Timestamp(6)
  updateTime DateTime? @updatedAt @map("update_time") @db.Timestamp(6)
}
```

- `@default(now())` - 自动填充当前 UTC 时间
- `@updatedAt` - 自动更新为当前 UTC 时间
- `@db.Timestamp(6)` - 微秒精度

### 3. 后端 API

NestJS 自动将 Date 对象序列化为 ISO 8601 格式：

```typescript
// Prisma 返回
{ createTime: Date }

// API 响应（自动序列化）
{ "createTime": "2026-01-16T08:30:00.000Z" }
```

特殊场景手动处理：

```typescript
// 导出 Excel 时转本地时间
createTime: user.createTime?.toLocaleString('zh-CN')

// 需要 ISO 字符串时
loginTime: user.loginTime?.toISOString()
```

### 4. 前端显示

使用 `web/src/utils/format.ts` 中的工具函数：

```typescript
import { formatDate, formatDateOnly, formatRelativeTime } from '@/utils/format'

// 完整日期时间：2026-01-16 16:30:00
formatDate(row.createTime)

// 仅日期：2026-01-16
formatDateOnly(row.createTime)

// 相对时间：5 分钟前
formatRelativeTime(row.createTime)
```

## 常见场景

### 列表展示

```vue
<template>
  <TableCell>{{ formatDate(row.createTime) }}</TableCell>
</template>

<script setup lang="ts">
import { formatDate } from '@/utils/format'
</script>
```

### 日期范围查询

前端传 ISO 字符串，后端直接用：

```typescript
// 前端
params: {
  beginTime: '2026-01-01T00:00:00.000Z',
  endTime: '2026-01-31T23:59:59.999Z',
}

// 后端
where: {
  createTime: {
    gte: new Date(dto.beginTime),
    lte: new Date(dto.endTime),
  }
}
```

### 导出 Excel

转换为本地时间字符串：

```typescript
{
  createTime: user.createTime 
    ? new Date(user.createTime).toLocaleString('zh-CN') 
    : ''
}
```

## 注意事项

### ✅ 正确做法

1. **存储**：始终使用 UTC
2. **传输**：使用 ISO 8601 格式（带 `Z` 后缀）
3. **显示**：前端统一用 `formatDate` 转本地时间
4. **比较**：使用 Date 对象或时间戳，不要比较字符串

### ❌ 避免的做法

1. 不要在数据库存本地时间
2. 不要手动拼接时间字符串
3. 不要在后端做时区转换（除非导出）
4. 不要用 `new Date().toString()`（格式不统一）

## 时区相关配置

### PostgreSQL

默认使用 UTC，无需配置。

### Node.js

服务器时区不影响，因为：
- `new Date()` 内部是 UTC 时间戳
- `@default(now())` 使用数据库时间

### 前端

`formatDate` 自动使用浏览器本地时区。

## FAQ

### Q: 为什么不用 TIMESTAMPTZ？

`TIMESTAMP` 和 `TIMESTAMPTZ` 在 PostgreSQL 中存储方式相同（都是 UTC），区别仅在于输入输出时是否自动转换。对于 Prisma 应用，两者效果一致。

### Q: Prisma Studio 显示的时间对吗？

对的，显示的是 UTC 时间。北京时间 = 显示时间 + 8 小时。

### Q: 如何调试时间问题？

```typescript
const date = new Date()
console.log('ISO:', date.toISOString())        // UTC
console.log('Local:', date.toLocaleString())   // 本地
console.log('Timestamp:', date.getTime())      // 毫秒时间戳
```
