# 前端页面交互与视觉规范

最后更新时间：2026-05-20

## 1. 文档定位

本文档是项目级全局开发规范，适用于后台管理端所有页面、组件和数据看板。

本规范用于沉淀“以后所有同类页面都应该怎么做”的通用准则，不记录单次故障复盘；单次问题、事故根因和防重复检查仍放在 `docs/问题解决/`。

## 2. 适用范围

- 表格页、列表页、看板页、统计页、详情页、弹窗和卡片区块。
- 缺陷管理、项目管理、系统管理、监控中心等所有业务模块。
- 项目仪表盘只是本规范的一个应用场景；风险色、数字色和刷新交互必须作为全局准则执行。

## 3. 页面级刷新按钮规范

### 3.1 基本原则

所有表格、列表、看板和统计数据页，都必须提供“刷新当前数据”的页面级操作。

刷新按钮属于项目开发规范，不属于单次问题防重复文档。后续新增或改造页面时，应默认把刷新能力纳入页面工具栏设计。

### 3.2 交互要求

- 刷新按钮必须只重新请求当前页面数据，禁止把“刷新当前数据”做成“刷新整个页面”。
- 点击刷新后必须保留当前筛选条件、分页条件、排序条件、展开状态和页面局部上下文。
- 禁止通过 `location.reload()`、浏览器整页刷新、路由重载或跳转中转页实现列表刷新。
- 刷新操作必须有 loading 反馈，loading 中按钮禁用，避免重复点击触发并发请求。
- 如果页面同时存在“刷新缓存”“刷新系统状态”等业务动作，必须和“刷新列表数据”区分命名、图标和说明。

### 3.3 视觉与位置

- 刷新入口放在页面 Header 右侧工具栏，靠近新增、导入、导出、清空等页面级操作。
- 默认使用轮廓按钮样式，避免抢占页面主操作视觉优先级。
- 图标使用刷新语义图标，左图标 + 文案“刷新”。
- 页面不应出现全局悬浮刷新按钮来替代表格/列表自身刷新入口。

### 3.4 组件复用

- 优先复用 `web/src/components/common/DataRefreshButton.vue`。
- 不要在每个页面重复手写刷新按钮结构、loading 图标和禁用逻辑。
- 如果刷新按钮能力扩展，例如 tooltip、快捷键、最近刷新时间，应优先扩展公共组件，而不是在单页面分叉实现。

### 3.5 验收标准

- 点击刷新后 URL 不变。
- 当前筛选、分页、排序和输入内容不被重置。
- 刷新过程中按钮显示 loading 并禁用。
- 页面 Header 工具栏内可见刷新入口。
- 没有使用整页刷新、路由重载或全局悬浮刷新按钮替代当前数据刷新。

## 4. 状态与风险语义色规范

### 4.1 基本原则

所有状态、风险、优先级、健康度、进度异常和统计数字，都必须使用全局统一的语义色。

颜色必须表达业务含义，不能因为“好看”随意给色；同一个语义在不同页面必须保持一致。

### 4.2 不只依赖颜色

- 风险和状态必须同时展示文字，例如“正常”“低风险”“高风险”“已延期”。
- 重要风险建议同时配合图标、标签样式或说明文本。
- 禁止只用红黄绿圆点表达状态，必须有可读文字，满足色弱用户和可访问性要求。
- 文本与背景对比度应满足常规可读性要求，深色模式下也必须可读。

### 4.3 全局语义色表

| 业务语义 | 推荐颜色 | 适用状态 | 说明 |
|---|---|---|---|
| 正常 / 健康 / 成功 | 绿色 `success` | 正常、健康、已完成、通过、按计划 | 表示无需干预或结果良好 |
| 低风险 / 轻微关注 | 青绿或蓝绿 `low-risk` | 低风险、轻微波动、可控风险 | 和“完全正常”区分，表示有风险但可控 |
| 处理中 / 信息 / 进行中 | 蓝色 `info` | 处理中、进行中、待发布、当前阶段 | 表示中性进展，不代表风险 |
| 关注 / 中风险 / 预警 | 黄色或琥珀色 `warning` | 关注、中风险、即将延期、质量波动 | 表示需要关注但尚未阻断 |
| 高风险 / 阻断 / 失败 | 红色 `danger` | 高风险、阻断 Bug、严重异常、失败 | 表示需要立即处理 |
| 已延期 / 逾期 / 超时 | 深红或红色 `overdue` | 已延期、逾期 N 天、超 SLA | 比普通高风险更强调时间失控 |
| 未知 / 无数据 / 草稿 | 灰色 `neutral` | 未知、无数据、草稿、未配置 | 表示状态不可判断或暂不评价 |

### 4.4 风险等级映射

项目管理、项目仪表盘、项目概览、需求列表、里程碑和统计卡片中的风险等级必须按以下语义映射：

| 风险值 | 展示文案 | 颜色语义 | 使用建议 |
|---|---|---|---|
| `healthy` / `normal` | 正常 | 绿色 `success` | 项目按计划推进，无明显阻塞 |
| `low` | 低风险 | 青绿或蓝绿 `low-risk` | 有轻微风险但不需要升级处理 |
| `watch` / `medium` | 关注 / 中风险 | 黄色或琥珀色 `warning` | 需要负责人持续跟进 |
| `high` / `risk` | 高风险 | 红色 `danger` | 需要管理层或负责人立即处理 |
| `delayed` / `overdue` | 已延期 | 深红或红色 `overdue` | 已超过计划时间或 SLA，需要解释原因和下一动作 |
| 空值 / 未配置 | 未配置 | 灰色 `neutral` | 提示补齐配置，不能伪装成正常 |

> 示例：项目仪表盘里的“正常”“低风险”“高风险”“已延期”必须有明确颜色区分，不能全部使用默认灰色描边标签。

### 4.5 数字指标颜色规范

数字颜色必须跟随业务含义，不允许所有数字统一使用同一种强调色。

| 数字类型 | 推荐颜色 | 示例 |
|---|---|---|
| 总数、中性数量 | 默认前景色 `foreground` | 项目总数、用户总数、需求总数 |
| 健康或正向数量 | 绿色 `success` | 正常推进项目数、已完成需求、已关闭 Bug、通过率优秀 |
| 低风险或轻微提醒数量 | 青绿或蓝绿 `low-risk` | 低风险项目、轻微波动指标 |
| 需要关注数量 | 黄色或琥珀色 `warning` | 风险项目、即将到期里程碑、待审核事项 |
| 严重异常数量 | 红色 `danger` | 高风险项目、阻断 Bug、失败任务 |
| 延期或超时数量 | 深红或红色 `overdue` | 已延期项目、逾期天数、超 SLA 工单 |
| 缺失或未知数量 | 灰色 `neutral` | 无数据、未配置、未知 |

### 4.6 Bug 严重程度、启停与版本状态映射

缺陷管理页除了流程状态和优先级，还必须语义化展示“严重程度”“启用/停用”和“版本状态”。

| 类型 | 值 | 展示文案 | 颜色语义 |
|---|---|---|---|
| Bug 严重程度 | `blocker` | 致命 | 深红 `overdue` |
| Bug 严重程度 | `critical` | 严重 | 红色 `danger` |
| Bug 严重程度 | `major` | 一般 | 黄色 `warning` |
| Bug 严重程度 | `minor` | 轻微 | 青绿 `low-risk` |
| 启停状态 | `0` / `enabled` | 启用 | 绿色 `success` |
| 启停状态 | `1` / `disabled` | 停用 | 灰色 `neutral` |
| Bug 版本状态 | `planning` | 规划中 | 蓝色 `info` |
| Bug 版本状态 | `testing` | 测试中 | 黄色 `warning` |
| Bug 版本状态 | `released` | 已发布 | 绿色 `success` |
| Bug 版本状态 | `archived` | 已归档 | 灰色 `neutral` |

实施要求：

- Bug 状态统一使用 `StatusBadge domain="bug"`。
- Bug 严重程度统一使用 `SeverityBadge`。
- Bug 优先级统一使用 `PriorityBadge`。
- 项目、模块、成员等启停状态统一使用 `StatusBadge domain="enabled"`。
- Bug 版本状态统一使用 `StatusBadge domain="bugVersion"`。
- 禁止在 缺陷列表、详情、统计卡片、重复 Bug 选择器中用纯文本或默认灰色标签展示上述状态。

### 4.7 趋势数字颜色

趋势数字不能简单按“上升绿色、下降红色”处理，必须先判断指标性质。

| 指标性质 | 上升 | 下降 |
|---|---|---|
| 越高越好 | 绿色 | 红色或琥珀色 |
| 越低越好 | 红色或琥珀色 | 绿色 |
| 中性变化 | 默认色或蓝色 | 默认色或蓝色 |

示例：

- Bug 关闭率上升是绿色，下降是红色或琥珀色。
- 高风险项目数量上升是红色，下降是绿色。
- 总项目数变化通常是中性，不应默认渲染成风险色。

### 4.8 进度条语义色规范

进度条必须和它表达的数据含义保持一致，不能所有进度条都使用默认主色。

| 进度条类型 | 颜色规则 | 示例 |
|---|---|---|
| 状态分布进度条 | 跟随对应状态语义色 | 缺陷看板“待确认”用 `warning`，“修复中”用 `info`，“已关闭”用 `success` |
| 严重程度/优先级进度条 | 跟随严重程度或优先级语义色 | 致命 Bug 用 `overdue`，严重 Bug 用 `danger`，轻微 Bug 用 `low-risk` |
| 风险/健康度进度条 | 跟随风险或健康度语义色 | 正常用 `success`，低风险用 `low-risk`，高风险用 `danger` |
| 中性分布进度条 | 使用 `neutral` 或 `info` | 负责人分布、项目占比等不直接代表风险的数据 |
| 完成率进度条 | 默认按进展语义，异常时升级为风险色 | 正常推进用 `info`/`success`，延期或阻断时用 `warning`/`danger`/`overdue` |

实施要求：

- 优先复用 `web/src/components/common/SemanticProgress.vue`，通过 `tone` 传入统一语义色。
- 语义色来源优先使用 `web/src/utils/semantic-styles.ts`，禁止在页面中散落硬编码进度条颜色。
- 进度条旁必须有文字、数值或标签说明，不能只依赖颜色传达业务状态。
- 禁止状态、风险、严重程度、优先级相关进度条继续使用默认 `primary` 主色。
- 深色模式下进度条轨道和填充色必须保持可辨识。

### 4.9 字段 Tag / Badge 使用规范

Tag / Badge 用于提高短字段的扫视效率，但必须先判断字段语义：表达“状态变化、风险、优先级、严重程度”的字段使用语义色；表达“对象归属、分类、人员、版本”的字段使用中性色或弱品牌色，避免把普通归属误读为风险或流程状态。

| 字段类型 | 推荐组件 / 样式 | 颜色规则 | 示例 |
|---|---|---|---|
| 流程状态 | `StatusBadge` | 使用对应业务域语义色 | Bug 状态、需求状态、迭代状态、里程碑状态 |
| 风险等级 | `RiskBadge` | 使用风险语义色 | 低风险、中风险、高风险、已延期 |
| 优先级 | `PriorityBadge` | 使用优先级语义色 | 紧急、高、中、低 |
| 严重程度 | `SeverityBadge` | 使用严重程度语义色 | 致命、严重、一般、轻微 |
| 成功 / 失败 | `SuccessFailBadge` | 成功绿色、失败红色，保持轻量 Tag | 操作日志、登录日志、任务执行结果 |
| 其它短语义字段 | `SemanticTag` | 通过 `tone` 选择统一语义色 | 数据库连接状态、系统内置、字典标签样式 |
| 项目 | `ProjectBadge` / `ProjectSelectValue` / `ProjectSelectOption` | 使用中性色或弱品牌色 | 迭代所属项目、需求所属项目、Bug 所属项目 |
| 模块 / 版本 | 中性 Badge，后续可沉淀 `ModuleBadge` / `VersionBadge` | 使用中性色；版本状态另用 `StatusBadge` | Bug 模块、影响版本、修复版本 |
| 人员 / 负责人 | 头像 + 姓名 Chip，后续可沉淀 `UserChip` | 使用中性色，不使用红黄绿风险色 | 负责人、提交人、验证人 |
| 角色 / 权限标识 | 中性或次级 Badge | 默认中性；安全等级等数值按业务规范另行定义 | 角色标识、权限编码、安全等级 |
| 纯文本描述 | 普通文本 | 不使用 Badge | 标题、目标、说明、备注 |

实施要求：

- 状态、风险、优先级、严重程度必须优先复用现有语义组件，禁止只用普通 `Badge variant="outline"` 或纯文本展示。
- 成功 / 失败类字段禁止在页面里手写 `Badge :variant="xxx ? 'default' : 'destructive'"`，必须使用 `SuccessFailBadge` 或 `SemanticTag`。
- 操作历史、动态流、通知、评论历史等面向用户的追溯信息必须展示业务可读文案：状态、动作、类型、环境等后端枚举必须通过统一映射转换，禁止直接显示 `pending_confirm`、`pending_verify`、`start_fix` 等内部 Key；未知纯技术枚举应显示“未知状态/未知操作”或补齐映射后再展示。
- Tag / Badge 是被动信息，不是操作入口；必须使用更轻量的胶囊样式、小号字号、浅底色和无明显 hover 效果，避免和按钮混淆。
- 可点击动作必须使用 `Button` 或 `SemanticActionButton`，不能伪装成 Tag；动作按钮应保留按钮高度、可点击 hover / focus 状态和更强视觉边界。
- 普通 `Badge` 的全局默认样式也必须保持标签形态，不得带按钮式 hover、阴影、强实色块或 `cursor-pointer`；如字段需要点击，应升级为 `Button`。
- 项目、模块、版本、人员属于对象归属或元数据，不得使用红 / 黄 / 绿状态色；除非字段本身表示状态，例如“版本状态：已发布”。
- 表格中项目类字段推荐使用 `ProjectBadge`，让项目归属具备稳定视觉锚点，同时通过项目编码辅助区分同名或近似项目。
- 下拉选择器中的项目类选项推荐使用 `ProjectSelectOption`：第一行项目标签，第二行展示阶段、负责人等辅助信息；不要把每条选项过度彩色化。
- 项目下拉选项的辅助信息必须展示业务可读文案：项目阶段等后端枚举必须先映射成中文标签，禁止把 `internal_test`、`release_ready`、`self_test` 等技术枚举值或内部 Key 直接展示给业务用户；无法识别的纯技术枚举默认不展示。
- 已选中的项目值推荐使用 `ProjectSelectValue` 或等效对象 Badge，而不是只显示普通文本；筛选器的“全部项目”仍保持普通文本，避免误认为某个真实项目。
- Badge 文案必须可读且可截断，长项目名、模块名、版本号应使用 `truncate`、Tooltip 或详情页承载完整信息，不能撑破表格或下拉面板。
- 不确定字段是否应该做成 Tag 时，先问“用户是否需要在一屏内快速区分它的类别 / 状态 / 归属”。如果只是长描述或业务标题，保持普通文本。

## 5. 数据看板与项目仪表盘规范

项目仪表盘、项目概览、统计看板等页面需要遵循以下全局准则：

- 顶部总览卡片必须区分中性指标、正向指标、风险指标和延期指标。
- 风险类卡片必须显示颜色、文案和可解释口径，避免管理层误判。
- 项目健康度、风险等级、延期状态必须使用统一语义色，不允许页面局部硬编码不同颜色。
- 关键数字必须能下钻或解释来源；不能只展示彩色大数字。
- 页面应显示最后更新时间，并提供手动刷新能力；如有自动刷新，也必须保留手动刷新。
- 风险列表必须显示风险原因和下一动作，不能只显示“高风险”。

## 6. 表格列对齐规范

### 6.1 基本原则

表格表头和单元格内容必须使用一致的主视觉锚点，禁止出现“表头左对齐、内容右对齐”或“表头居中、内容左对齐”的矛盾排版。

项目默认采用“表头跟随该列内容对齐”的规则：文本列左对齐、数值列右对齐、短标签和控件列居中、操作列按操作性质固定对齐。

### 6.2 对齐规则

| 列类型 | 表头对齐 | 内容对齐 | 示例 | 说明 |
|---|---|---|---|---|
| 长文本 / 标题 / 名称 / 描述 | 左对齐 | 左对齐 | 需求标题、Bug 标题、项目名、模块名 | 长字段不居中，便于阅读和扫视 |
| 人员 / 组织 / 负责人 | 左对齐 | 左对齐 | 负责人、创建人、部门 | 人名和组织属于文本信息 |
| 编号 / ID / 单号 | 左对齐 | 左对齐 | 需求编号、Bug 编号、工单号 | 即使含数字，也不做数学比较，按文本处理 |
| 日期 / 时间 | 左对齐 | 左对齐 | 计划完成、创建时间、更新时间 | 本项目默认左对齐；若全站统一固定宽度日期列，可另行评估居中 |
| 状态 / 风险 / 优先级 / 严重程度标签 | 居中 | 居中 | 状态、风险、优先级、严重程度 | 短标签居中更稳定，表头必须同步居中 |
| 数量 / 金额 / 百分比 / 天数 / 评分 | 右对齐 | 右对齐 | 数量、Bug 关闭率、逾期天数 | 便于纵向比较大小，数字建议使用 `tabular-nums` |
| 进度列 | 右对齐或居中 | 与表头一致 | 进度条 + 百分比 | 如果百分比靠右，表头也右对齐；如果整体居中，表头也居中 |
| 复选框 / 图标 / 开关 | 居中 | 居中 | 多选框、启停开关、图标状态 | 控件型列按控件中心对齐 |
| 快捷操作列 | 左对齐 | 左对齐 | 提交、通过、驳回、流转状态 | 按钮数量不固定，用户从左到右寻找下一步动作 |
| 操作列 | 右对齐 | 右对齐 | 查看、编辑、删除 | 固定为最后一列，用于管理动作收口 |

### 6.3 快捷操作列与操作列

- 表格最后一列统一命名为“操作”，只放查看、编辑、删除、配置、详情等管理动作。
- 当前项目“操作”列的主功能是编辑和删除；如果当前用户没有任何编辑、删除或其他管理动作权限，必须隐藏整列，连“操作”表头也不要显示。
- 如果一行存在状态流转、审批、指派、验证、关闭、重开等高频快捷按钮，必须新增独立“快捷操作”列。
- “快捷操作”列放在“操作”列前面，左对齐；“操作”列永远放在最后面，右对齐。
- 如果当前用户对该表没有任何快捷操作权限，必须隐藏整列，连“快捷操作”表头也不要显示。
- 禁止把快捷流转按钮和编辑/删除混在同一个“操作”列里，避免用户误点破坏性动作。
- 是否展示“操作”列或“快捷操作”列，应按当前用户权限集合和当前表格能力先计算列级可见性；不要只在单元格内部用权限指令隐藏按钮后留下空表头、空列或整列 `-`。
- 只有当该列对当前用户整体可见、但某一行因为状态或数据条件没有可执行动作时，才在该行显示 `-` 或“无可用操作”，不要留空。

### 6.4 权限按钮可见性

- 所有受权限控制的业务按钮必须和“编辑”“删除”保持一致：当前用户没有对应权限时，按钮必须直接不渲染。
- 禁止把无权限业务按钮做成禁用态、隐藏文字但保留图标、保留空按钮占位或只在点击后提示无权限。
- 适用按钮包括但不限于：新增、编辑、删除、批量删除、批量启停、导入、导出、刷新缓存、清空日志、强退、重置密码、执行一次、状态切换、解锁、成员维护和流程流转。
- 确认弹窗里的确认按钮同样属于权限按钮，例如确认删除、确认清空、确认强退、确认重置密码、确认批量启停、确认执行一次；无权限时必须不渲染确认按钮，而不是只隐藏入口按钮。
- 公共 `Button`、`DropdownMenuItem` 和 `AlertDialogAction` 已支持按当前路由、按钮文案、图标、确认弹窗标题自动推断权限；新增复杂或动态动作时仍应优先显式传 `permission`，避免“确定/保存”等泛化文案误判。
- 公共 `Button` 无权限时会直接不渲染实际按钮；如果是 `as-child` 触发器或下拉触发器，也必须隐藏整个触发入口，不能留下空按钮、空图标或可点击空区域。
- 业务下拉菜单项、导出格式菜单项、附件上传入口、附件移除入口也属于权限按钮；无权限时必须不渲染真实菜单项或按钮。
- 纯 UI 触发器、无事件原生按钮、取消/关闭/退出登录/放弃更改/全选/取消全选/个人中心/主题切换等普通交互，应显式标记 `data-permission-neutral` 或由公共组件识别为权限中性，避免被“确认/删除”等文案误判为业务权限按钮。
- 状态开关、上传组件、附件移除、导出组件、确认弹窗和快捷操作组件都必须把权限向内部确认按钮、下拉菜单项或触发按钮透传；不能只隐藏外层入口而让内部确认按钮仍可触发受限接口。
- 业务 API 可在前端封装层增加二次权限兜底，用于避免页面漏按钮时误发请求；但这只是体验和防误触兜底，不能替代后端 `@RequirePermission`。
- 搜索、重置筛选、刷新当前数据、展开/收起、取消、关闭弹窗等不访问受限业务接口的普通交互按钮，不按权限按钮处理。
- 前端隐藏只用于界面体验；所有受限业务接口仍必须保留后端权限校验。

### 6.5 快捷操作按钮语义色

快捷操作按钮必须根据动作结果或风险语义着色，帮助用户一眼识别下一步动作，但不能为了“好看”随意彩色化。

| 动作类型 | 推荐语义色 | 示例动作 |
|---|---|---|
| 推进 / 开始 / 排期 | `info` | 提交评审、排期、开始、开始开发 |
| 评审 / 测试 / 需要关注 | `warning` | 开始评审、提交测试、进入测试、暂停 |
| 正向完成 / 通过 / 发布 | `success` | 评审通过、验收通过、发布、完成、达成 |
| 关闭 / 归档类中性收口 | `neutral` | 关闭、归档 |
| 延期 / 超时 | `overdue` | 延期、超 SLA |
| 取消 / 驳回 / 重开 / 负向动作 | `danger` | 取消、驳回、重新打开 |

实施要求：

- 优先复用 `web/src/components/common/SemanticActionButton.vue`。
- Bug 状态流转动作（确认有效、驳回、指派、开始修复、提交验证、验证通过、重新打开等）也必须使用 `SemanticActionButton`，不能继续使用统一灰色 `secondary` 或普通 `outline`。
- 颜色必须来自动作语义映射，不允许在业务页面散落硬编码按钮颜色。
- 快捷操作按钮必须保留明确文字，不能只靠颜色或图标表达含义。
- 破坏性管理动作，例如删除，仍保留在最后“操作”列，不应混入“快捷操作”列。
- 快捷操作按钮必须和状态 Tag 明显区分：按钮使用 `SemanticActionButton` 的动作按钮样式，保留可点击高度、实色或更强边框、hover / focus 状态；状态 Tag 使用 `StatusBadge` 的浅底胶囊样式，禁止复用按钮形态。
- 同一表格中状态列与快捷操作列相邻时，优先采用“状态 Tag 轻、动作按钮重”的层级；不要只靠文案区分“已完成”和“完成”。

### 6.6 实施要求

- 数字和百分比建议使用 `tabular-nums`，避免数字宽度跳动。
- 长字段表头不要为了“看起来居中”而居中；应通过缩短文案、设置列宽、tooltip 或换行解决表头过长问题。
- 表头和内容必须同步设置对齐 class，例如内容 `text-right` 时表头也必须 `text-right`。
- 公共表格组件已提供基于表头文案的默认对齐推断：`操作` 默认右对齐、`快捷操作` 默认左对齐、状态/风险/优先级等短标签默认居中、数量/百分比/耗时/进度等数值列默认右对齐；业务页面显式传入 `text-left/text-center/text-right` 时以业务页面为准。
- 状态标签如果内容居中展示，表头也应居中。
- 同一张表内，同类列的对齐方式必须一致。
- 新增表格或改造表格时，必须检查表头与内容对齐是否一致。

### 6.7 看板状态颜色要求

- 看板列头数量标签必须跟随列语义色，例如“已完成”为成功色，“暂缓/异常”为危险色。
- 看板卡片内的状态标签必须使用 `StatusBadge`，禁止全部使用灰色 `outline`。
- 看板卡片内的优先级标签必须使用 `PriorityBadge`，紧急/高优先级应明显区别于低优先级。
- 项目名称、负责人、时间等普通信息可以继续使用中性样式，避免把非状态信息误标成风险色。

## 7. 长列表与看板滚动容器规范

### 7.1 基本原则

表格、列表、看板列、卡片组、动态流、历史记录和详情子列表都必须预设“数据很多”的展示方案。页面不能因为子项过多而无限向下拉长，导致用户找不到页面头部、筛选器、操作区或其他区块。

只要同一区块内可能出现超过 5 条同类记录，开发时就必须评估限高、分页、虚拟列表、折叠或内部滚动方案，不能等真实数据变多后再补。

### 7.2 适用场景

以下场景默认按长列表处理：

- 项目看板列中的需求卡片、缺陷卡片、任务卡片。
- 仪表盘、项目概览、统计页中的当前处理、历史完成、未处理、项目动态、风险列表、行动建议。
- 详情弹窗中的附件、评论、操作历史、关联需求、关联缺陷。
- 任何卡片内的子项列表、时间线、审批记录、通知记录。
- 表格页中嵌套的展开行子列表。

### 7.3 容器高度与滚动要求

- 卡片内容区列表默认设置最大高度，例如 `max-h-[30rem] overflow-y-auto overscroll-contain pr-3`。
- 看板列应使用固定或视口相关高度，例如 `max-h-[calc(100vh-16rem)]` 或页面可用高度计算，并让列内部滚动。
- 横向看板可以横向滚动列，但每一列内部仍必须纵向限高滚动，禁止所有卡片把整页无限撑高。
- 弹窗内长列表必须在弹窗内容区内滚动，禁止把弹窗撑出视口；优先使用 `max-h-[60vh] overflow-y-auto` 或项目已有滚动弹窗组件。
- 长弹窗禁止把 `DialogContent` 本身作为滚动容器，例如禁止 `DialogContent class="max-h-[90vh] overflow-y-auto"` 这类写法。必须采用“弹窗容器 `overflow-hidden` + 固定 Header/Footer + 中间内容区 `min-h-0 overflow-y-auto`”的结构，确保标题、关闭按钮、保存/关闭按钮不会随内容滚走。
- 长弹窗推荐使用 `grid max-h-[90vh] grid-rows-[auto_1fr_auto] gap-0 overflow-hidden p-0`；如果没有底部操作区，可使用 `grid-rows-[auto_1fr]`。仅使用 `max-h-[90vh] flex flex-col` 但未给容器 `overflow-hidden`、内容区 `min-h-0` 的写法不合格，内容过长或嵌套编辑器时仍可能把头部挤走。
- 弹窗 Header/Footer 必须放在滚动容器外，并使用明确的背景和分隔线，例如 `border-b bg-background px-6 py-4 pr-14`、`border-t bg-background px-6 py-4`。
- 弹窗关闭按钮必须始终可见并与 Header 共享固定区域；Header 右侧要预留关闭按钮空间（例如 `pr-14`），避免标题或状态标签和关闭按钮重叠；关闭按钮的无障碍文案必须本地化为中文“关闭”，不得暴露 `Close`。
- 内部滚动容器应使用 `overscroll-contain`，避免滚到底后连带外层页面滚动造成定位丢失。
- 滚动容器右侧预留内边距，例如 `pr-3`，避免滚动条遮挡按钮、标签或文字。

### 7.4 长弹窗固定头尾专项规范

当弹窗内容可能超过一屏，或弹窗内包含详情、表单、附件、评论、操作历史、富文本编辑器、树形选择器等长内容时，必须按“头部固定、底部固定、中间内容滚动”的结构实现。不能让用户因为向下滚动而找不到弹窗标题、关闭按钮、保存按钮或取消按钮。

#### 7.4.1 强制结构

有底部操作区的长弹窗必须使用以下结构：

```vue
<DialogContent class="grid max-h-[90vh] grid-rows-[auto_1fr_auto] gap-0 overflow-hidden p-0">
  <DialogHeader class="border-b bg-background px-6 py-4 pr-14">
    <DialogTitle>弹窗标题</DialogTitle>
    <DialogDescription>弹窗说明</DialogDescription>
  </DialogHeader>

  <div class="min-h-0 overflow-y-auto overscroll-contain px-6 py-4">
    <!-- 长内容、表单、历史记录、附件列表等 -->
  </div>

  <DialogFooter class="border-t bg-background px-6 py-4">
    <Button variant="outline">取消</Button>
    <Button>确定</Button>
  </DialogFooter>
</DialogContent>
```

没有底部操作区的详情弹窗，可以使用：

```vue
<DialogContent class="grid max-h-[90vh] grid-rows-[auto_1fr] gap-0 overflow-hidden p-0">
  <DialogHeader class="border-b bg-background px-6 py-4 pr-14">
    ...
  </DialogHeader>

  <div class="min-h-0 overflow-y-auto overscroll-contain px-6 py-4">
    ...
  </div>
</DialogContent>
```

#### 7.4.2 禁止写法

以下写法禁止用于长弹窗：

```vue
<!-- 错误：整个弹窗滚动，头部、关闭按钮和底部操作会被滚走 -->
<DialogContent class="max-h-[90vh] overflow-y-auto">
  ...
</DialogContent>

<!-- 错误：看似分区，但缺少 overflow-hidden 和 min-h-0，长内容仍可能把头尾挤出视口 -->
<DialogContent class="max-h-[90vh] flex flex-col">
  <DialogHeader />
  <div class="flex-1 overflow-y-auto">...</div>
  <DialogFooter />
</DialogContent>
```

#### 7.4.3 交互与可访问性要求

- Header/Footer 必须在滚动容器外，且使用 `bg-background`，避免内容滚动时视觉穿透。
- Header 右侧必须预留关闭按钮空间，例如 `pr-14`，避免标题、状态标签或项目标签和关闭按钮重叠。
- 右上角关闭按钮必须始终可见；无障碍文案必须使用中文“关闭”，不得暴露英文 `Close`。
- 中间内容滚动区必须使用 `min-h-0 overflow-y-auto overscroll-contain`，避免 Grid/Flex 子项默认最小高度导致滚动失效。
- 底部主操作按钮必须始终可见；表单类弹窗滚到底部前也必须能点击“取消/保存/确定”。
- 弹窗内如果还有子列表、代码块、日志块等二级滚动区，二级滚动区也应限制高度，避免产生难以操作的多层无限滚动。

#### 7.4.4 验收标准

- 模拟内容超过一屏并滚动到底部时，标题、关闭按钮和底部操作仍然可见。
- 弹窗自身不超出视口，不出现页面外层跟随滚动导致弹窗位置丢失。
- 鼠标滚轮在内容区滚动到底后，不应明显带动背景页面滚动。
- 深色模式下 Header/Footer 分隔线清晰，不与内容区混在一起。
- 移动端或窄屏下关闭按钮、底部按钮仍可点击，不被滚动条或安全区遮挡。

### 7.5 数据量很大时的优先级

- 10 条以内：可使用限高 + 内部滚动。
- 10 到 100 条：优先分页、分组、筛选、折叠或加载更多；不能只靠页面无限滚动。
- 超过 100 条或频繁刷新：优先虚拟列表或服务端分页，避免一次性渲染大量 DOM。
- 管理类表格仍优先使用分页；看板列和卡片流可使用列内滚动 + 加载更多。

### 7.6 操作与可用性要求

- 长列表区顶部标题、筛选器、项目选择器、刷新按钮和主要操作按钮必须保持可达，不能被长列表推到很远的位置。
- 长列表子项如果有详情能力，必须提供可见“详情”按钮；表格行可支持双击进入详情，但不能只依赖双击。
- 空状态、加载状态、错误状态必须在滚动容器内正常展示，不能因为限高导致不可见。
- 列表项标题过长时应使用换行、`line-clamp`、tooltip 或详情弹窗处理，不能把卡片横向撑破。
- 移动端下长列表优先单列布局，滚动容器高度要根据视口调整，避免出现多个难以操作的嵌套滚动区。

### 7.7 写入前检查

开发或改造表格、列表、看板、统计卡片和详情弹窗前，必须检查：

- 是否存在同一区块可能超过 5 条记录的子列表。
- 是否存在看板列、历史记录、项目动态、评论、附件等天然会增长的数据。
- 是否存在长弹窗；如有必须套用第 7.4 节“固定头尾 + 中间滚动”结构。
- 是否需要分页、加载更多、折叠、虚拟列表或服务端限制 `pageSize`。
- 当前自有代码文件行数是否接近 450 行；接近时先拆组件，不要把长列表逻辑继续堆在页面文件里。

### 7.8 写入后验收

- 模拟 20 条以上数据时，页面整体高度不会被单个列表无限撑长。
- 关键操作区、筛选区和刷新按钮仍然容易找到。
- 列表内部可以顺畅滚动，滚动条不遮挡内容和按钮。
- 长弹窗滚动到底时，弹窗标题、关闭按钮和底部操作按钮仍然固定可见。
- 空状态、加载状态、详情按钮、双击详情等交互在限高容器内仍正常。
- 深色模式、移动端和窄屏下没有出现横向撑破或按钮不可点。

## 8. 配置化与组件化要求

### 8.1 统一映射

状态、风险、优先级和数字指标颜色应优先通过统一映射配置维护，例如：

- 风险等级映射。
- 状态标签映射。
- 指标卡片语义映射。
- 趋势升降语义映射。

禁止在多个页面散落写死颜色 class，避免同一状态在不同页面颜色不一致。

### 8.2 已落地公共能力与扩展方向

当前已落地的公共能力：

- `web/src/components/common/DataRefreshButton.vue`：页面级数据刷新按钮。

- `web/src/components/common/RiskBadge.vue`：风险等级标签。
- `web/src/components/common/StatusBadge.vue`：需求、Bug、迭代、里程碑、项目阶段等状态标签。
- `web/src/components/common/PriorityBadge.vue`：需求或 Bug 优先级标签。
- `web/src/components/common/SeverityBadge.vue`：Bug 严重程度标签。
- `web/src/components/common/MetricCard.vue`：统计数字卡片。

## 9. 表单可用性规范

### 9.1 基本原则

表单不能只依赖 placeholder 或默认选项表达字段含义。placeholder 只用于输入示例或短提示，不能替代字段标签、字段用途说明和选择后果说明。

所有新增、编辑、筛选、导入配置类表单，都必须让用户在不阅读代码、不询问开发人员的情况下理解：这个字段是什么、是否必填、默认值代表什么、不填写会产生什么后果。

### 9.2 字段标签与必填标识

- 每个可编辑字段必须有可见 `Label`，禁止只放一个输入框或下拉框。
- 必填字段必须在标签上标明 `*` 或“必填”，并在保存前给出明确错误提示。
- 可选字段建议在标签中标明“可选”，尤其是负责人、日期、备注、配置项等容易影响后续流程的字段。
- 弹窗标题只说明表单类型，例如“新增需求”；字段本身仍必须逐项标明名称。

### 9.3 帮助说明与默认值解释

### 9.2.1 Tooltip 与说明文案规范

- Tooltip、说明 icon、帮助文案、指标解释文案必须优先使用“用户能直接理解的人话”，不要写字段名、数据库列名、接口参数名、内部变量名或实现细节。
- 文案长度应尽量控制在 1 到 2 句，优先解释“这是什么”“为什么会这样”“怎么理解当前数字”，不要堆砌背景实现。
- 如果指标存在多个口径，Tooltip 必须明确说明当前展示口径，例如“这是负责人手动维护的整体进度”或“这是已完成需求占全部需求的比例”。
- Tooltip 只讲用户关心的重点，不要出现类似 `bug_project.progress`、`requirementDoneRate`、DTO 字段名、SQL 计算表达式等技术术语。
- Tooltip 响应要足够快，默认应即时或接近即时出现，避免用户悬停后长时间无反馈。
- 同类页面中的同类说明文案必须保持一致；项目看板、项目概览、仪表盘、详情弹窗不要各写各的说法。
- 如果一段说明超过 2 句，优先改写成更短的人话，而不是继续往 Tooltip 里堆技术解释。


下列字段必须提供短说明，不能只显示“暂不指定”“请选择”“年月日”：

| 字段类型 | 必须说明的内容 | 示例 |
|---|---|---|
| 负责人 / 处理人 / 审核人 | 该角色负责什么、不指定时谁处理或何时再补 | “需求负责人负责需求澄清与验收；暂不指定时可稍后补充。” |
| 开发 / 测试 / 评审角色 | 该角色在哪个流程节点使用 | “开发负责人用于排期后承接开发任务。” |
| 日期 / 时间 | 日期含义、是否影响提醒/逾期/统计 | “计划开始时间用于排期统计，可先留空。” |
| 分类 / 类型 / 状态 | 选项影响后续什么规则 | “需求分类影响统计口径，不影响优先级。” |
| 默认值 / 暂不指定 | 默认值代表的业务结果 | “暂不指定表示当前不绑定负责人，不会自动分派。” |
| 风险 / 优先级 / 严重程度 | 选择标准与处理建议 | “高优先级表示需要优先排期处理。” |

### 9.4 placeholder 使用规则

- placeholder 只能写“输入示例”或“未输入时的短提示”，不能承担字段名称职责。
- 日期控件必须配套标签和说明，不能只显示浏览器默认的“年/月/日”。
- 下拉框如果已有默认值，也仍需要标签和说明；默认值不是说明。
- 不允许连续多个“暂不指定”字段没有上下文说明，必须分别说明“暂不指定负责人”和“暂不指定开发负责人”的业务含义。

### 9.5 表单布局与阅读顺序

- 弹窗表单优先使用“标签 + 控件 + 帮助说明”的纵向结构。
- 两列表单中，同一行左右字段应属于同一层级信息，例如“项目 / 分类”“优先级 / 负责人”“计划开始 / 计划完成”。
- 复杂表单应按“基础信息、人员分工、计划时间、详细说明”分组，而不是把所有控件平铺。
- 长文本字段必须有标签，placeholder 只提示填写内容，例如“描述用户场景、业务价值或问题背景”。

### 9.6 公共组件与复用要求

表单字段应优先复用 `web/src/components/common/FormFieldBlock.vue` 这类统一结构，避免每个页面手写不同的标签、必填标识和说明样式。

推荐结构：

```vue
<FormFieldBlock
  label="开发负责人"
  field-id="xxx-developer"
  optional
  description="负责排期后的开发承接；暂不指定时需求仍可保存，但后续需补齐承接人。"
>
  <Select v-model="form.developerId">...</Select>
</FormFieldBlock>
```

使用要求：

- `label` 必须是用户可理解的业务名称，不写数据库字段名或内部变量名。
- `field-id` 应与内部控件 `id` 对应，便于无障碍读取和点击标签聚焦。
- 必填字段使用 `required`，可选但会影响流程的字段使用 `optional`。
- `description` 写业务用途、默认值后果或不填写影响，不重复 placeholder。
- 页面内出现 3 个以上同类字段时，必须优先使用统一组件，禁止继续复制零散 `Label + p` 结构。

### 9.7 写入前检查

改造或新增表单前必须检查：

- 是否每个字段都有可见标签。
- 是否每个“暂不指定”“默认”“空值”都有业务后果说明。
- 是否每个日期/时间字段说明了用途。
- 是否可选字段与必填字段区分清楚。
- 是否存在同类字段可复用统一文案，优先复用统一文案而不是临时硬写。

### 9.8 写入后验收

- 用户只看界面即可理解每个字段用途。
- 表单不存在连续无标签输入框。
- 表单不存在多个相同“暂不指定”但无法区分含义的下拉框。
- 表单不存在只有“年/月/日”而不知道日期用途的日期控件。
- 深色模式下标签、说明、错误提示可读。

### 9.9 表单控件焦点态规范

表单控件必须有清晰的选中 / 聚焦状态，让用户明确知道当前正在编辑哪个字段。

适用控件：

- `Input`：文本、数字、日期、时间等输入框。
- `Textarea`：多行文本输入。
- `SelectTrigger`：下拉选择器触发区域。
- `Checkbox` / `RadioGroupItem`：勾选和单选控件。

实现要求：

- 聚焦时必须有可见状态变化，优先使用 1px 级别边框色变化，避免边框过粗造成视觉噪音。
- 焦点色使用全局 `border-ring` / `ring` 语义，不允许页面内硬编码临时颜色。
- 文本输入、日期输入、多行文本和下拉选择器优先只改变边框色；如需焦点环，应使用 `ring-inset` 或等效内描边，避免弹窗、滚动容器、表格单元格中左右边缘被裁切。
- Checkbox / Radio 这类小尺寸控件可以保留轻量 `ring-1 ring-inset`，确保键盘焦点可见。
- 下拉选择器在键盘聚焦和展开状态都必须保持同一套焦点态。
- 禁止通过给单个页面额外加 `border-*` / `ring-*` 来修补焦点态；应优先修改公共控件。

当前焦点态统一落点：

- `web/src/components/ui/input/Input.vue`
- `web/src/components/ui/textarea/Textarea.vue`
- `web/src/components/ui/select/SelectTrigger.vue`
- `web/src/components/ui/checkbox/Checkbox.vue`
- `web/src/components/ui/radio-group/RadioGroupItem.vue`

验收标准：

- 在角色管理新增 / 编辑弹窗、需求管理弹窗、Bug 提交 / 编辑弹窗中，点击任意输入控件时能清楚看到当前字段。
- 控件位于弹窗最左侧、最右侧或滚动区域边界时，焦点色不被裁切。
- 深色模式下焦点态仍可见且不刺眼。

## 10. 写入前检查

开发或改造前端页面前，需要检查：

- 当前页面是否属于表格、列表、看板或统计页；如果是，是否已有刷新入口。
- 当前页面是否展示状态、风险、优先级、健康度、异常、延期或趋势数字。
- 是否已有公共组件或统一配置可复用。
- 是否存在同类页面的既有实现，优先保持一致，而不是复制一套新样式。

## 11. 写入后验收

- 表格/列表/看板页具备页面级刷新按钮，且刷新不重置上下文。
- 风险、状态和优先级标签具备颜色、文字和必要说明。
- 项目仪表盘等看板中的关键数字按业务含义使用语义色。
- 同一业务语义在不同页面颜色一致。
- 深色模式下文字可读。
- 色弱用户不依赖颜色也能理解状态。
- 页面没有散落大量硬编码颜色；新增颜色规则应进入统一配置或公共组件。


## 12. 菜单业务域归属规范

菜单归属必须以业务域和用户心智为准，不能只按技术复用关系或页面实现路径归类。

- 缺陷列表、我的缺陷、提交缺陷、缺陷看板等围绕缺陷生命周期的页面，必须归属 `缺陷管理` 菜单。
- 项目概览、需求管理、迭代计划、里程碑、项目看板等围绕项目计划和交付推进的页面，必须归属 `项目管理` 菜单。
- 项目配置、模块管理、版本管理虽然服务 Bug 录入，但本项目已作为项目基础资料统一归入 `项目管理`；新增类似配置页时应先判断其是否为跨模块基础资料。
- 旧路径可保留兼容重定向，但菜单、权限文档、工作台默认路径和种子脚本必须指向新的业务域路径。
- 新增菜单时必须同步检查路由、数据库种子、角色权限文档、工作台路径选项和默认首页配置，避免同一页面在多个业务域重复出现。
