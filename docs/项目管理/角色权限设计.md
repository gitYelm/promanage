# Bug 反馈系统角色权限设计

最后更新时间：2026-05-17
当前状态：已收敛为一套规范业务角色；技术标识保留 `bug_*`，前台显示名称去除 Bug 前缀，重复通用角色已停用/逻辑删除
前置文档：[需求分析.md](./需求分析.md)

## 1. 设计目标

本设计用于明确 Bug 反馈系统第一版的用户角色、权限点、数据可见范围、状态流转权限和典型使用场景。

系统基于现有 `RBAC Admin Pro` 的 RBAC 权限体系扩展，不单独重做权限系统。后端可继续使用现有 `@RequirePermission('模块:资源:动作')` 的权限校验风格，前端通过菜单权限和按钮权限控制页面入口与操作按钮。

## 2. 权限设计原则

1. **角色职责清晰**：提交、确认、分派、修复、验证、关闭分别由不同角色负责。
2. **数据范围受控**：内部系统也需要控制项目可见范围，避免所有成员默认看到所有项目。
3. **流程完整优先**：第一版以完整闭环流程为目标，不为了快速上线省略关键状态和操作记录。
4. **操作必须留痕**：创建、编辑、分派、状态变化、评论、附件上传和删除都需要记录操作历史。
5. **权限点细分到按钮**：菜单只控制入口，关键业务操作必须独立配置按钮权限。
6. **超级管理员兜底**：现有 `admin` 角色保留全部权限，用于系统维护和异常处理。

## 3. 角色清单

| 角色标识 | 角色名称 | 角色定位 | 是否第一版必需 |
|---|---|---|---|
| admin | 超级管理员 | 拥有系统所有权限 | 是，沿用现有角色 |
| bug_project_owner | 项目负责人 | 管理项目内缺陷、成员、分派和统计 | 是 |
| bug_product_owner | 产品负责人 | 确认缺陷有效性，调整优先级和项目模块归属 | 是 |
| bug_reviewer | 审核人员 | 审核缺陷、驳回或分派给开发 | 是 |
| bug_developer | 开发人员 | 接收分派、修复缺陷、提交验证 | 是 |
| bug_tester | 测试人员 | 提交缺陷、复现、验证、关闭或重新打开 | 是 |
| bug_submitter | 提交人 | 提交并跟踪本人缺陷 | 是 |
| bug_operator | 运营客服 | 可选历史角色；暂无岗位时不启用 | 否，默认停用 |
| bug_viewer | 观察者 | 可选只读历史角色；如需只读可另行启用 | 否，默认停用 |
| system_admin | 系统管理员 | 维护用户、角色、菜单、字典等系统配置 | 沿用现有角色 |

> 说明：技术角色标识继续使用 `bug_*` 以保持权限兼容，但角色列表显示名称统一为“项目负责人、产品负责人、审核人员、开发人员、测试人员、提交人、管理层”。不要再创建 `developer`、`tester`、`reviewer` 这类重复系统角色；项目内成员角色仍使用 `owner/product/reviewer/developer/tester/viewer`。

## 4. 角色职责说明

### 4.1 超级管理员

职责：

1. 系统兜底管理。
2. 维护全部项目和权限配置。
3. 处理异常数据和误操作。
4. 拥有所有菜单、按钮和数据权限。

### 4.2 项目负责人

职责：

1. 查看负责项目下的全部 Bug。
2. 管理项目、模块、版本和项目成员。
3. 调整 Bug 优先级、严重程度和负责人。
4. 协调超期或长期未处理 Bug。
5. 查看项目统计数据。

### 4.3 产品负责人

职责：

1. 确认 Bug 是否有效。
2. 判断 Bug 类型、优先级和严重程度。
3. 补充或修正项目、模块、版本信息。
4. 将确认有效的 Bug 推进到待分派或已分配。
5. 驳回非 Bug、重复、暂不处理的问题。

### 4.4 Bug 审核人员

职责：

1. 查看负责项目的待确认和重新打开 Bug。
2. 判断 Bug 是否有效、是否重复、是否暂不处理。
3. 驳回无效 Bug 或要求提交人补充信息。
4. 将有效 Bug 指派给项目内开发人员。
5. 确保 developer 在分派前不可见、不可处理 tester 新提交缺陷。

### 4.5 开发人员

职责：

1. 查看分派给自己的 Bug。
2. 接收任务并开始修复。
3. 添加评论、补充附件或询问信息。
4. 修复完成后填写修复说明并提交验证。
5. 对重新打开的 Bug 继续处理。

### 4.6 测试人员

职责：

1. 提交缺陷。
2. 复现和补充问题信息。
3. 验证开发修复结果。
4. 验证通过后关闭 Bug。
5. 验证不通过时重新打开 Bug。

### 4.7 提交人

职责：

1. 在后台提交缺陷。
2. 查看自己提交或参与评论的 Bug。
3. 补充复现步骤、截图、日志、录屏等信息。
4. 在允许状态下编辑本人提交的 Bug。

### 4.8 运营/客服

职责：

1. 内部代客户或业务方提交缺陷。
2. 补充业务背景、截图和联系方式。
3. 跟进处理状态。

第一版如果没有运营/客服参与，可以暂不分配用户，保留角色用于后续代提交流程。

### 4.9 Bug 观察者

职责：

1. 只读查看被授权项目的 缺陷列表、详情和统计。
2. 关注项目质量趋势、当前未处理数量和处理进度。
3. 不参与确认、分派、修复、验证、关闭等流程动作。
4. 适用于老板、领导、跨团队协作方或只需旁观进度的成员。

## 5. 数据可见范围

### 5.1 推荐数据范围

| 角色 | 可见数据范围 |
|---|---|
| 超级管理员 | 全部项目、全部 Bug |
| 系统管理员 | 默认不自动拥有 Bug 业务数据权限，除非额外授予 |
| 项目负责人 | 自己负责或参与的项目下全部 Bug |
| 产品负责人 | 自己参与项目下全部 Bug |
| 审核人员 | 自己参与审核项目下全部 Bug，重点处理待确认、重新打开和待分派 Bug |
| 开发人员 | 仅分派给自己的 Bug；未分派前不可见 tester 新提交缺陷 |
| 测试人员 | 自己提交、自己负责验证、自己参与评论或被授权项目下需验证的 Bug |
| 提交人 | 自己提交的 Bug；自己被 @ 或参与评论的 Bug；作为项目 viewer 时可只读查看项目 Bug |
| 运营/客服 | 自己提交或被授权项目下的 Bug |
| 观察者 | 被授权项目下的只读 Bug 和统计数据 |

### 5.2 项目成员机制建议

建议第一版增加“项目成员”概念，否则多项目场景下数据范围会混乱。

项目成员建议包含：

| 字段 | 说明 |
|---|---|
| 项目 | 成员所属项目 |
| 用户 | 项目成员用户 |
| 项目内角色 | owner、product、reviewer、developer、tester、viewer |
| 是否默认负责人 | 可选，用于模块或项目默认指派 |
| 状态 | 启用、停用 |

### 5.3 数据权限实现建议

后端查询 Bug 时建议按以下顺序判断：

1. 如果是超级管理员，返回全部数据。
2. 如果拥有项目级管理权限，返回其负责或 active 成员项目的数据。
3. 如果是项目负责人、产品负责人或项目 reviewer，返回其负责/审核项目的数据。
4. 如果是开发人员，只返回分派给自己的 Bug，以及自己参与评论的 Bug。
5. 如果是测试/提交人，只返回本人提交、本人验证、本人参与评论或被 @ 的 Bug。
6. 删除或敏感操作必须再次校验按钮权限和数据归属。

### 5.4 系统角色与项目数据权限边界

系统角色只决定菜单和按钮能力，不能直接决定项目数据范围。

| 项 | 规则 |
|---|---|
| `bug_project_owner` | 表示具备项目负责人功能权限；不是“所有项目负责人”，不能自动看全部项目 |
| `bug_product_owner` | 表示具备产品负责人功能权限；只能看自己参与的项目 |
| `bug_reviewer` | 表示具备审核功能权限；只能看自己参与审核的项目 |
| `pm_executive` | 管理层角色可看仪表盘入口；是否看全部项目必须通过专门权限或项目授权控制 |
| 项目负责人字段 | `BugProject.ownerId` 是项目数据归属的重要依据 |
| 项目成员表 | `bug_project_member.status = '0'` 才表示 active 成员授权 |

因此项目、模块、版本、项目仪表盘、项目概览、需求、迭代和里程碑等所有项目管理接口，普通业务用户的项目可见范围必须统一为：

1. 自己是 `BugProject.ownerId` 的项目；
2. 自己是 active 项目成员的项目；
3. 超级管理员才可见全部项目。

即使用户拥有 `pm:project:view`、`pm:requirement:view` 或 `pm:executive-dashboard:view`，也只能进入页面和执行查询动作，不能绕过项目归属看到未授权项目。

菜单展示还必须遵循“非空目录”规则：系统为角色绑定按钮权限时可能会自动带出上级目录，但 `/getRouters` 只应返回拥有可见子菜单的目录。若某个一级目录下没有任何可显示页面菜单，例如项目负责人没有系统管理页面权限，则不能显示空的“系统管理”菜单。

### 5.5 角色安全等级与上下级分配规则

角色安全等级用于解决“谁能创建、编辑、分配、选择、指派、维护谁”的问题，不能直接复用 `roleSort`。`roleSort` 只适合做角色列表展示排序，不能代表权限高低；例如 `common_user` 的排序值可能小于业务角色，但不代表它权限更高。

当前安全等级作为 `sys_role.security_level` 字段持久化保存，后端以 `RoleSecurityService` 和 `SystemRoleSecurityService` 为统一兜底。以下表格是内置角色的默认初始化等级：

| 角色标识 | 安全等级 | 说明 |
|---|---:|---|
| `admin` | 1000 | 超级管理员，系统兜底 |
| `system_admin` | 900 | 系统配置管理员 |
| `monitor_admin` | 800 | 监控/运维类管理员 |
| `pm_executive` | 700 | 管理层/项目仪表盘 |
| `bug_project_owner` | 600 | 项目负责人 |
| `bug_product_owner` | 550 | 产品负责人 |
| `bug_reviewer` | 520 | 审核人员 |
| `bug_developer` | 400 | 开发人员 |
| `bug_tester` | 350 | 测试人员 |
| `common_user` | 200 | 普通用户 |
| `bug_submitter` | 100 | 提交人 |

规则：

1. 一个用户绑定多个系统角色时，安全等级取最高值；例如 `project_owner` 同时拥有 `bug_project_owner` 和 `bug_product_owner` 是允许的，实际等级按项目负责人计算。
2. 非 `admin` 操作者不能选择、指派、加入、移除或停用安全等级高于自己的用户；这里的“更高权限”包括但不限于超级管理员、系统管理员、管理层和更高业务角色。
3. 项目负责人不能直接变更 `BugProject.ownerId`；如需负责人交接，应由超级管理员或后续专门的负责人交接流程执行。
4. 项目成员维护、模块默认负责人、缺陷指派、需求负责人/开发/测试负责人、迭代负责人、里程碑负责人都必须做后端字段级校验，不能只依赖前端下拉框。
5. 缺陷和需求创建/编辑时显式传入 `projectId`，必须属于当前用户可见项目；关联的模块、版本、需求、迭代、里程碑必须属于同一项目。
6. 系统用户管理中，非 `admin` 不能给用户分配高于自己的角色，用户列表/详情/导出也不能暴露权限高于自己的用户，不能编辑、停用、删除、重置密码权限高于自己的用户。
   通用导出任务入口同样必须校验 `system:user:export`，并按操作者安全等级过滤可导出的用户。
7. 系统角色管理中，非 `admin` 的角色列表/详情不能暴露权限高于自己的角色，不能编辑、停用、删除权限高于自己的角色；新增/编辑角色时只能设置小于等于自己安全等级的 `securityLevel`。
8. 系统角色授权菜单时，非 `admin` 只能授予自己已经拥有的菜单/按钮权限，不能通过编辑低等级角色间接创造更高系统权限；菜单授权树也只返回操作者自己已有的菜单/按钮。
9. 菜单授权树、工作台配置的角色选项、详情、编辑和删除也必须接入同一套角色安全等级，不能绕过角色管理页面单独维护高等级角色配置。
10. `/system/user/getInfo` 返回给前端的 `roles`、`roleList` 和 `permissions` 只允许基于启用且未删除的角色生成，避免停用角色继续让前端显示菜单或按钮。
11. 通用导出任务必须按模块校验导出权限：用户导出需要 `system:user:export`，缺陷统计导出需要 `bug:statistics:export`；导出任务详情、下载、删除只能操作本人创建的任务，下载时还要重新校验当前是否仍拥有对应模块导出权限。
12. 缺陷统计导出必须按操作者可见项目过滤数据，不能因为拥有统计导出按钮就导出全量项目缺陷。
13. 前端候选人下拉必须传明确场景参数，例如 `assignContext=projectOwner/bugAssignee/requirementOwner/moduleAssignee`、`projectId` 和 `assignableOnly=true`；但最终权限以服务端校验结果为准。

项目成员角色与系统角色映射如下：

| 项目成员角色 | 必须具备的系统角色 |
|---|---|
| `owner` | `bug_project_owner` |
| `product` | `bug_product_owner` |
| `reviewer` | `bug_reviewer` |
| `developer` | `bug_developer` |
| `tester` | `bug_tester` |
| `viewer` | 暂不强制系统业务角色，仅允许只读 |

## 6. 菜单设计

当前采用两个一级菜单分工：`缺陷管理` 负责工单处理入口，`项目管理` 负责项目进度、需求、迭代、配置和统计入口。

| 菜单 | 类型 | 建议路由 | 权限标识 |
|---|---|---|---|
| 缺陷管理 | 目录 | `/bug` | - |
| 缺陷列表 | 菜单 | `/bug/tickets` | `bug:ticket:list` |
| 我的缺陷 | 菜单 | `/bug/my` | `bug:ticket:my` |
| 提交缺陷 | 菜单 | `/bug/create` | `bug:ticket:add` |
| 缺陷看板 | 菜单 | `/bug/statistics` | `bug:statistics:view` |
| 项目管理 | 目录 | `/project-management` | - |
| 项目仪表盘 | 菜单 | `/project-management/executive-dashboard` | `pm:executive-dashboard:view` |
| 项目概览 | 菜单 | `/project-management/overview` | `pm:project:view` |
| 需求管理 | 菜单 | `/project-management/requirements` | `pm:requirement:view` |
| 迭代计划 | 菜单 | `/project-management/iterations` | `pm:iteration:view` |
| 里程碑 | 菜单 | `/project-management/milestones` | `pm:milestone:view` |
| 项目看板 | 菜单 | `/project-management/board` | `pm:dashboard:view` |
| 项目配置 | 菜单 | `/project-management/projects` | `bug:project:list` |
| 模块管理 | 菜单 | `/project-management/modules` | `bug:module:list` |
| 版本管理 | 菜单 | `/project-management/versions` | `bug:version:list` |

> 说明：现有系统的菜单类型通常包含目录、菜单、按钮。以上是建议的目录和菜单，按钮权限见下一节。

## 10. 关键使用场景

### 10.1 测试提交并关闭 Bug

1. 测试人员在后台提交缺陷，状态为 `待确认`。
2. 产品负责人确认有效，状态变为 `已确认`。
3. 产品负责人或项目负责人指派开发，状态变为 `已分配`。
4. 开发人员开始修复，状态变为 `修复中`。
5. 开发人员修复完成并填写修复说明，状态变为 `待验证`。
6. 测试人员验证通过，状态变为 `已关闭`。

### 10.2 提交信息不足

1. 提交人创建 Bug，但复现步骤不完整。
2. 产品或测试在评论区要求补充信息。
3. 提交人补充截图、日志或录屏。
4. 产品或测试再次确认后继续流转。

### 10.3 验证不通过重新打开

1. Bug 进入 `待验证`。
2. 测试人员验证后发现问题仍存在。
3. 测试人员填写不通过原因并重新打开。
4. 项目负责人或产品负责人重新分派。
5. 开发人员继续修复。

### 10.4 重复 Bug 处理

1. 产品或测试发现新 Bug 与已有 Bug 重复。
2. 将状态标记为 `重复问题`。
3. 必须关联原 Bug 编号。
4. 重复 Bug 不进入开发处理流程。

## 11. 边界规则

### 11.1 提交人编辑规则

建议第一版规则：

1. `待确认` 状态下，提交人可以编辑本人提交的 Bug。
2. 进入 `已确认` 后，提交人不能直接修改核心字段，只能通过评论补充信息。
3. 已关闭 Bug 不允许编辑，只允许按权限重新打开。

### 11.2 删除规则

建议第一版不提供普通删除能力。

1. 超级管理员可以逻辑删除 Bug。
2. 项目负责人默认不能删除 Bug，避免破坏流程追踪。
3. 删除必须记录操作日志。
4. 被删除 Bug 不在普通列表显示，但可在后台审计中保留。

### 11.3 附件删除规则

1. 上传人可在 Bug 未关闭前删除自己上传的附件。
2. 项目负责人和超级管理员可以删除项目内附件。
3. 附件删除必须记录历史。
4. 已关闭 Bug 的附件原则上不允许删除，除非管理员处理敏感信息。

### 11.4 评论删除规则

1. 普通用户只能删除本人评论。
2. 进入 `已关闭` 后不建议允许普通用户删除评论。
3. 管理员删除评论需要记录原因。

## 12. 与现有 RBAC 系统的对接建议

现有项目已包含：

1. 用户管理
2. 角色管理
3. 菜单管理
4. 部门管理
5. `SysRole`、`SysMenu`、`SysRoleMenu` 等权限模型
6. `@RequirePermission()` 后端权限装饰器

建议实现方式：

1. 在种子数据中新增 缺陷管理菜单、页面权限和按钮权限。
2. 为新增角色分配默认权限。
3. 后端 Controller 使用 `@RequirePermission('bug:ticket:list')` 等权限标识。
4. 前端页面根据权限控制按钮显示。
5. 数据范围不要只依赖菜单权限，必须在服务层按项目成员、负责人、提交人过滤。

## 14. 已确认决策

1. 第一版采用“系统 RBAC 角色 + 项目成员角色”双层权限模型。
2. 单独创建 `bug_reviewer` 系统角色，并在项目成员角色中启用 `reviewer`。
3. `bug_operator`、`bug_viewer` 当前作为历史可选角色默认停用；如确有运营客服或只读观察场景，再按需启用。
4. 开发人员默认只能看到分派给自己的 Bug；tester 新提交且未分派的 Bug 不通知、不展示给 developer。
5. 提交人允许在 `待确认` 状态编辑本人 Bug；进入审核后只能通过评论和附件补充。
6. 删除 Bug 采用逻辑删除，仅超级管理员拥有删除权限。
7. `roleSort` 不作为安全等级；角色上下级统一走 `sys_role.security_level`，所有负责人/成员候选人选择都必须校验“不能选择权限高于自己的用户”。
8. 系统用户/角色/菜单授权/工作台配置管理也必须接入同一套安全等级，避免有系统菜单权限的人反向维护更高权限账号或角色，或给角色授予自己并不拥有的菜单权限。

## 17. 角色管理页面等级展示规则

为避免把“显示顺序”误解成“权限等级”，角色管理页面必须显式展示后端安全等级：

1. 后端角色列表和详情接口统一返回 `securityLevel`。
2. 角色列表展示“安全等级”列，显示数值和说明，例如 `600 / 项目负责人`。
3. 新增/编辑弹窗、角色详情/权限预览都展示“安全等级”；新增/编辑时允许填写安全等级。
4. 新增/编辑角色时可以填写安全等级，但后端必须强制校验：非 `admin` 只能设置小于等于自己安全等级的值。
5. 角色安全等级保存到 `sys_role.security_level`；内置角色通过迁移初始化默认等级，后续以数据库字段为准。
