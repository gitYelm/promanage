# Bug 反馈系统角色权限设计

最后更新时间：2026-05-15
当前状态：已按 tester/reviewer/developer 闭环权限更新
前置文档：[需求分析.md](./需求分析.md)

## 1. 设计目标

本设计用于明确 Bug 反馈系统第一版的用户角色、权限点、数据可见范围、状态流转权限和典型使用场景。

系统基于现有 `RBAC Admin Pro` 的 RBAC 权限体系扩展，不单独重做权限系统。后端可继续使用现有 `@RequirePermission('模块:资源:动作')` 的权限校验风格，前端通过菜单权限和按钮权限控制页面入口与操作按钮。

## 2. 权限设计原则

1. **角色职责清晰**：提交、确认、分派、修复、验证、关闭分别由不同角色负责。
2. **数据范围受控**：内部系统也需要控制项目可见范围，避免所有成员默认看到所有项目。
3. **流程完整优先**：第一版以完整闭环流程为目标，不为了快速上线省略关键状态和操作记录。
4. **操作必须留痕**：创建、编辑、分派、状态变化、评论、附件上传和删除都需要记录操作历史。
5. **权限点细分到按钮**：菜单只控制入口，关键业务操作必须独立配置按钮权限。
6. **超级管理员兜底**：现有 `admin` 角色保留全部权限，用于系统维护和异常处理。

## 3. 角色清单

| 角色标识 | 角色名称 | 角色定位 | 是否第一版必需 |
|---|---|---|---|
| admin | 超级管理员 | 拥有系统所有权限 | 是，沿用现有角色 |
| bug_project_owner | 项目负责人 | 管理项目内 Bug、成员、分派和统计 | 是 |
| bug_product_owner | 产品负责人 | 确认 Bug 有效性，调整优先级和项目模块归属 | 是 |
| bug_reviewer / 项目内 reviewer | 审核人员 | 审核 Bug、驳回或分派给开发 | 是，可用项目内角色实现 |
| bug_developer | 开发人员 | 接收分派、修复 Bug、提交验证 | 是 |
| bug_tester | 测试人员 | 提交 Bug、复现、验证、关闭或重新打开 | 是 |
| bug_submitter | 提交人 | 提交并跟踪本人 Bug | 是 |
| bug_operator | 运营/客服 | 内部代提交、补充信息、协助跟进 | 可选 |
| system_admin | 系统管理员 | 维护用户、角色、菜单、字典等系统配置 | 沿用现有角色 |

> 说明：同一个用户可以拥有多个角色，例如测试人员也可以是提交人，项目负责人也可能兼任产品负责人。

## 4. 角色职责说明

### 4.1 超级管理员

职责：

1. 系统兜底管理。
2. 维护全部项目和权限配置。
3. 处理异常数据和误操作。
4. 拥有所有菜单、按钮和数据权限。

### 4.2 项目负责人

职责：

1. 查看负责项目下的全部 Bug。
2. 管理项目、模块、版本和项目成员。
3. 调整 Bug 优先级、严重程度和负责人。
4. 协调超期或长期未处理 Bug。
5. 查看项目统计数据。

### 4.3 产品负责人

职责：

1. 确认 Bug 是否有效。
2. 判断 Bug 类型、优先级和严重程度。
3. 补充或修正项目、模块、版本信息。
4. 将确认有效的 Bug 推进到待分派或已分配。
5. 驳回非 Bug、重复、暂不处理的问题。

### 4.4 开发人员

职责：

1. 查看分派给自己的 Bug。
2. 接收任务并开始修复。
3. 添加评论、补充附件或询问信息。
4. 修复完成后填写修复说明并提交验证。
5. 对重新打开的 Bug 继续处理。

### 4.5 测试人员

职责：

1. 提交 Bug。
2. 复现和补充问题信息。
3. 验证开发修复结果。
4. 验证通过后关闭 Bug。
5. 验证不通过时重新打开 Bug。

### 4.6 提交人

职责：

1. 在后台提交 Bug。
2. 查看自己提交或参与评论的 Bug。
3. 补充复现步骤、截图、日志、录屏等信息。
4. 在允许状态下编辑本人提交的 Bug。

### 4.7 运营/客服

职责：

1. 内部代客户或业务方提交 Bug。
2. 补充业务背景、截图和联系方式。
3. 跟进处理状态。

第一版如果没有运营/客服参与，可以暂不单独创建该角色，使用提交人角色即可。

## 5. 数据可见范围

### 5.1 推荐数据范围

| 角色 | 可见数据范围 |
|---|---|
| 超级管理员 | 全部项目、全部 Bug |
| 系统管理员 | 默认不自动拥有 Bug 业务数据权限，除非额外授予 |
| 项目负责人 | 自己负责或参与的项目下全部 Bug |
| 产品负责人 | 自己参与项目下全部 Bug |
| 开发人员 | 仅分派给自己的 Bug；未分派前不可见 tester 新提交 Bug |
| 测试人员 | 自己提交、自己负责验证、自己参与评论或被授权项目下需验证的 Bug |
| 提交人 | 自己提交的 Bug；自己被 @ 或参与评论的 Bug |
| 运营/客服 | 自己提交或被授权项目下的 Bug |

### 5.2 项目成员机制建议

建议第一版增加“项目成员”概念，否则多项目场景下数据范围会混乱。

项目成员建议包含：

| 字段 | 说明 |
|---|---|
| 项目 | 成员所属项目 |
| 用户 | 项目成员用户 |
| 项目内角色 | 项目负责人、产品、审核人员、开发、测试、观察者等 |
| 是否默认负责人 | 可选，用于模块或项目默认指派 |
| 状态 | 启用、停用 |

### 5.3 数据权限实现建议

后端查询 Bug 时建议按以下顺序判断：

1. 如果是超级管理员，返回全部数据。
2. 如果拥有项目级管理权限，返回其负责项目的数据。
3. 如果是项目负责人、产品负责人或项目 reviewer，返回其负责/审核项目的数据。
4. 如果是开发人员，只返回分派给自己的 Bug，以及自己参与评论的 Bug。
5. 如果是测试/提交人，只返回本人提交、本人验证、本人参与评论或被 @ 的 Bug。
6. 删除或敏感操作必须再次校验按钮权限和数据归属。

## 6. 菜单设计

当前采用两个一级菜单分工：`Bug 管理` 负责工单处理入口，`项目管理` 负责项目进度、需求、迭代、配置和统计入口。

| 菜单 | 类型 | 建议路由 | 权限标识 |
|---|---|---|---|
| Bug 管理 | 目录 | `/bug` | - |
| Bug 列表 | 菜单 | `/bug/tickets` | `bug:ticket:list` |
| 我的 Bug | 菜单 | `/bug/my` | `bug:ticket:my` |
| 提交 Bug | 菜单 | `/bug/create` | `bug:ticket:add` |
| 项目管理 | 目录 | `/project-management` | - |
| 老板驾驶舱 | 菜单 | `/project-management/executive-dashboard` | `pm:executive-dashboard:view` |
| 项目概览 | 菜单 | `/project-management/overview` | `pm:project:view` |
| 需求管理 | 菜单 | `/project-management/requirements` | `pm:requirement:view` |
| 迭代计划 | 菜单 | `/project-management/iterations` | `pm:iteration:view` |
| 里程碑 | 菜单 | `/project-management/milestones` | `pm:milestone:view` |
| 项目看板 | 菜单 | `/project-management/board` | `pm:dashboard:view` |
| 项目配置 | 菜单 | `/project-management/projects` | `bug:project:list` |
| 模块管理 | 菜单 | `/project-management/modules` | `bug:module:list` |
| 版本管理 | 菜单 | `/project-management/versions` | `bug:version:list` |
| Bug 看板 | 菜单 | `/project-management/statistics` | `bug:statistics:view` |

> 说明：现有系统的菜单类型通常包含目录、菜单、按钮。以上是建议的目录和菜单，按钮权限见下一节。

## 9. 状态流转权限

| 当前状态 | 动作 | 下一状态 | 推荐可操作角色 | 必填说明 |
|---|---|---|---|---|
| 待确认 | 确认为有效 | 已确认 | 审核人员、项目负责人、产品负责人、超管 | 确认说明可选 |
| 待确认 | 驳回 | 已驳回 | 审核人员、项目负责人、产品负责人、超管 | 驳回原因建议必填 |
| 待确认 | 标记无法复现 | 无法复现 | 审核人员、项目负责人、产品负责人、超管 | 原因建议必填 |
| 待确认 | 标记重复 | 重复问题 | 审核人员、项目负责人、产品负责人、超管 | 关联重复 Bug 必填 |
| 已确认 | 指派负责人 | 已分配 | 产品负责人、项目负责人、超管 | 负责人必填 |
| 已分配 | 开始修复 | 修复中 | 被指派开发、超管 | 可选说明 |
| 修复中 | 提交验证 | 待验证 | 被指派开发、超管 | 修复说明建议必填 |
| 待验证 | 验证通过 | 已关闭 | 测试、审核人员、超管 | 验证说明建议必填 |
| 待验证 | 验证不通过 | 重新打开 | 测试、审核人员、超管 | 不通过原因建议必填 |
| 重新打开 | 重新分派 | 已分配 | 审核人员、项目负责人、产品负责人、超管 | 负责人必填 |
| 已关闭 | 重新打开 | 重新打开 | 测试、提交人受限、审核人员、超管 | 重开原因建议必填 |
| 已确认/已分配 | 暂不处理 | 暂不处理 | 审核人员、项目负责人、产品负责人、超管 | 暂缓原因建议必填 |
| 暂不处理 | 重新进入处理 | 已确认 | 审核人员、项目负责人、产品负责人、超管 | 说明可选 |

## 10. 关键使用场景

### 10.1 测试提交并关闭 Bug

1. 测试人员在后台提交 Bug，状态为 `待确认`。
2. 产品负责人确认有效，状态变为 `已确认`。
3. 产品负责人或项目负责人指派开发，状态变为 `已分配`。
4. 开发人员开始修复，状态变为 `修复中`。
5. 开发人员修复完成并填写修复说明，状态变为 `待验证`。
6. 测试人员验证通过，状态变为 `已关闭`。

### 10.2 提交信息不足

1. 提交人创建 Bug，但复现步骤不完整。
2. 产品或测试在评论区要求补充信息。
3. 提交人补充截图、日志或录屏。
4. 产品或测试再次确认后继续流转。

### 10.3 验证不通过重新打开

1. Bug 进入 `待验证`。
2. 测试人员验证后发现问题仍存在。
3. 测试人员填写不通过原因并重新打开。
4. 项目负责人或产品负责人重新分派。
5. 开发人员继续修复。

### 10.4 重复 Bug 处理

1. 产品或测试发现新 Bug 与已有 Bug 重复。
2. 将状态标记为 `重复问题`。
3. 必须关联原 Bug 编号。
4. 重复 Bug 不进入开发处理流程。

## 11. 边界规则

### 11.1 提交人编辑规则

建议第一版规则：

1. `待确认` 状态下，提交人可以编辑本人提交的 Bug。
2. 进入 `已确认` 后，提交人不能直接修改核心字段，只能通过评论补充信息。
3. 已关闭 Bug 不允许编辑，只允许按权限重新打开。

### 11.2 删除规则

建议第一版不提供普通删除能力。

1. 超级管理员可以逻辑删除 Bug。
2. 项目负责人默认不能删除 Bug，避免破坏流程追踪。
3. 删除必须记录操作日志。
4. 被删除 Bug 不在普通列表显示，但可在后台审计中保留。

### 11.3 附件删除规则

1. 上传人可在 Bug 未关闭前删除自己上传的附件。
2. 项目负责人和超级管理员可以删除项目内附件。
3. 附件删除必须记录历史。
4. 已关闭 Bug 的附件原则上不允许删除，除非管理员处理敏感信息。

### 11.4 评论删除规则

1. 普通用户只能删除本人评论。
2. 进入 `已关闭` 后不建议允许普通用户删除评论。
3. 管理员删除评论需要记录原因。

## 12. 与现有 RBAC 系统的对接建议

现有项目已包含：

1. 用户管理
2. 角色管理
3. 菜单管理
4. 部门管理
5. `SysRole`、`SysMenu`、`SysRoleMenu` 等权限模型
6. `@RequirePermission()` 后端权限装饰器

建议实现方式：

1. 在种子数据中新增 Bug 管理菜单、页面权限和按钮权限。
2. 为新增角色分配默认权限。
3. 后端 Controller 使用 `@RequirePermission('bug:ticket:list')` 等权限标识。
4. 前端页面根据权限控制按钮显示。
5. 数据范围不要只依赖菜单权限，必须在服务层按项目成员、负责人、提交人过滤。

## 13. 第一版推荐默认角色授权

| 角色 | 默认授权建议 |
|---|---|
| 超级管理员 | 全部 Bug 权限 |
| 项目负责人 | 负责项目内全部 Bug 操作、项目成员、模块、版本、统计 |
| 产品负责人 | Bug 确认、驳回、分派、编辑核心信息、评论、附件、统计 |
| 开发人员 | 仅查看分派给自己的 Bug、开始修复、提交验证、评论、附件；不可看到 tester 刚提交的待确认 Bug |
| 测试人员 | 提交、确认、验证、关闭、重新打开、评论、附件 |
| 提交人 | 提交、查看本人相关 Bug、待确认时编辑本人 Bug、评论、附件 |
| 运营/客服 | 代提交、查看本人或授权项目 Bug、评论、附件 |

## 14. 待确认问题

进入业务流程图和页面设计前，建议确认以下问题：

1. 是否同意第一版增加“项目成员”机制？建议：同意。

6. 删除 Bug 是否采用逻辑删除？建议：采用逻辑删除。
