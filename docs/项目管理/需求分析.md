# Bug 反馈系统需求分析

最后更新时间：2026-05-16
当前状态：一期实现中；已确认启用站内通知、bug_reviewer/bug_viewer 和 tester/reviewer/developer 闭环权限

## 1. 背景与目标

本项目基于现有 `RBAC Admin Pro` 后台管理系统扩展，实现一个面向内部团队使用的 Bug 反馈与处理系统。

系统目标是为内部产品、研发、测试、运营等团队提供统一的问题反馈、分派、处理、验证和关闭流程，减少 Bug 信息分散、责任不清、状态不可追踪的问题。

## 2. 已确认需求

| 项目 | 结论 |
|---|---|
| 使用对象 | 内部团队使用 |
| 项目范围 | 支持多个项目 |
| 提交入口 | 后台登录后提交 |
| 附件能力 | 需要支持截图、日志、录屏等附件；截图上传时需要支持在线标注编辑 |
| 通知提醒 | 第一版启用站内通知；tester 提交后通知 reviewer/负责人，developer 仅在被分派后收到通知 |
| 状态流程 | 采用完整状态流转流程 |
| 第一版目标 | 流程完整、权限清晰、可追踪 |

## 3. 建议系统定位

本系统定位为内部团队 Bug 闭环管理平台，主要覆盖：

1. 内部成员提交 Bug。
2. 产品或负责人确认 Bug 有效性。
3. 指派开发人员处理。
4. 开发完成后流转给测试验证。
5. 测试通过后关闭 Bug。
6. 全过程保留操作历史、评论和附件。

## 4. 需求范围

### 4.1 第一版范围内

第一版建议实现以下能力：

1. 多项目管理
2. 项目模块管理
3. 版本管理
4. Bug 提交
5. Bug 列表查询
6. Bug 详情查看
7. Bug 状态流转
8. Bug 指派负责人
9. 评论协作
10. 截图、日志、录屏等附件上传
11. 截图在线标注编辑
12. 操作历史记录
13. 基础统计看板
14. 基于 RBAC 的菜单和按钮权限控制
15. 站内通知：按下一步处理人精准通知，不对开发人员泛发待确认 Bug

### 4.2 第一版范围外

第一版暂不实现：

1. 外部客户免登录反馈入口
2. 邮件、短信、飞书、钉钉、企业微信等外部通知
3. SLA 自动超期提醒
4. 自动去重或智能相似 Bug 推荐
5. 移动端独立应用
6. 与 Git、CI/CD、工单系统的深度集成
7. AI 自动分析、AI 自动修复、本地 Codex 接入、AI 修复包
8. 公开反馈社区或用户投票机制


### 4.3 二期规划范围：AI 修复能力

AI 修复能力已确认放入二期，一期不实现。

二期目标是：在开发人员补充清晰修复说明后，由本机 AI Runner 调用本地 Codex，对管理员配置过的本地项目执行代码分析、隔离修复、测试检查、Diff 回传和人工确认应用。

二期规划包括：

1. AI 分析：只读分析 Bug 根因和预计影响文件。
2. AI 修复：在隔离工作区调用本地 Codex 修复代码。
3. AI 修复包：将多个同模块、同文件或同流程 Bug 合并为一次修复。
4. 本地项目映射：Bug 项目与本机项目路径配置化绑定。
5. AI Runner：本机服务负责领取任务、调用 Codex、回传日志和 Diff。
6. 冲突检测：修复结果应用前必须检查 patch 是否可合并。
7. 审计日志：保存 Prompt、命令、日志、修改文件和结果摘要。

二期详细规划见：[二期AI修复规划.md](./二期AI修复规划.md) 和 [AI修复架构设计.md](../架构文档/AI修复架构设计.md)。

## 5. 用户角色

| 角色 | 主要职责 |
|---|---|
| 系统管理员 | 维护用户、角色、菜单、字典、系统配置 |
| 提交人 | 在后台提交 Bug，查看自己提交或参与的 Bug |
| 产品负责人 | 确认 Bug 是否有效，调整优先级和所属项目模块 |
| 审核人员 | 处理待确认审核池，驳回无效问题或分派给开发 |
| 开发人员 | 接收分派，处理 Bug，填写修复说明 |
| 测试人员 | 复现 Bug，验证修复结果，关闭或重新打开 Bug |
| 项目负责人 | 查看项目内所有 Bug，协调处理进度，关注统计数据 |
| 观察者 | 只读查看被授权项目 Bug 和统计，不参与流程动作 |

## 6. 核心业务流程

### 6.1 标准状态流程

```text
待确认 → 已确认 → 已分配 → 修复中 → 待验证 → 已关闭
```

### 6.2 异常状态

```text
已驳回
无法复现
重复问题
暂不处理
重新打开
```

### 6.3 状态说明

| 状态 | 含义 | 典型操作人 |
|---|---|---|
| 待确认 | Bug 已提交，等待确认是否有效 | 提交人、产品、测试 |
| 已确认 | Bug 被确认有效，等待分派 | 产品、项目负责人 |
| 已分配 | 已指定处理人 | 产品、项目负责人 |
| 修复中 | 开发正在处理 | 开发人员 |
| 待验证 | 开发已修复，等待测试验证 | 开发人员、测试人员 |
| 已关闭 | 验证通过或确认无需继续处理 | 测试人员、项目负责人 |
| 已驳回 | 非 Bug、信息无效或不受理 | 产品、测试、项目负责人 |
| 无法复现 | 当前信息不足或无法稳定复现 | 测试、开发 |
| 重复问题 | 与已有 Bug 重复 | 产品、测试、项目负责人 |
| 暂不处理 | 问题有效但暂缓处理 | 产品、项目负责人 |
| 重新打开 | 已关闭或待验证后发现仍存在问题 | 测试、提交人、项目负责人 |

## 7. 功能需求

### 7.1 项目管理

用于维护多个项目，作为 Bug 归属的基础数据。

字段建议：

| 字段 | 说明 |
|---|---|
| 项目名称 | 必填，项目显示名称 |
| 项目标识 | 必填，用于编号或接口标识 |
| 项目负责人 | 可选，默认负责人 |
| 项目描述 | 可选 |
| 状态 | 启用、停用 |

### 7.2 项目模块管理

用于维护每个项目下的业务模块。

字段建议：

| 字段 | 说明 |
|---|---|
| 所属项目 | 必填 |
| 模块名称 | 必填 |
| 默认负责人 | 可选 |
| 排序 | 可选 |
| 状态 | 启用、停用 |

### 7.3 版本管理

用于记录 Bug 出现或修复对应的版本。

字段建议：

| 字段 | 说明 |
|---|---|
| 所属项目 | 必填 |
| 版本号 | 必填 |
| 版本名称 | 可选 |
| 发布时间 | 可选 |
| 状态 | 规划中、测试中、已发布、已归档 |

### 7.4 Bug 提交

后台登录用户可以提交 Bug。

字段建议：

| 字段 | 是否必填 | 说明 |
|---|---|---|
| 标题 | 是 | 简明描述问题 |
| 所属项目 | 是 | 支持多个项目 |
| 所属模块 | 是 | 项目下的模块 |
| 发现版本 | 否 | Bug 出现的版本 |
| 问题类型 | 是 | 功能异常、界面问题、性能问题、兼容问题等 |
| 严重程度 | 是 | 致命、严重、一般、轻微 |
| 优先级 | 是 | 紧急、高、中、低 |
| 问题描述 | 是 | 详细描述问题 |
| 复现步骤 | 是 | 按步骤描述如何复现 |
| 期望结果 | 是 | 正常情况下应该如何表现 |
| 实际结果 | 是 | 当前实际表现 |
| 运行环境 | 否 | 生产、测试、预发、本地等 |
| 设备信息 | 否 | 浏览器、系统、设备型号等 |
| 附件 | 否 | 截图、日志、录屏；截图支持上传前或上传后在线标注编辑 |

提交后默认状态为 `待确认`。

### 7.5 Bug 列表

列表需要支持查询、筛选、排序和分页。

展示字段建议：

| 字段 | 说明 |
|---|---|
| Bug 编号 | 系统生成 |
| 标题 | Bug 标题 |
| 项目 | 所属项目 |
| 模块 | 所属模块 |
| 状态 | 当前处理状态 |
| 严重程度 | 影响范围 |
| 优先级 | 处理优先级 |
| 提交人 | 创建人 |
| 负责人 | 当前处理人 |
| 创建时间 | 提交时间 |
| 更新时间 | 最近更新时间 |

筛选条件建议：

1. 关键词
2. 所属项目
3. 所属模块
4. 状态
5. 严重程度
6. 优先级
7. 提交人
8. 负责人
9. 创建时间范围
10. 更新时间范围

### 7.6 Bug 详情

详情页应包含：

1. 基础信息
2. 问题描述
3. 复现步骤
4. 期望结果和实际结果
5. 附件列表与预览
6. 截图标注编辑结果预览
7. 当前状态
7. 当前负责人
8. 状态流转操作区
9. 评论讨论区
10. 操作历史记录
11. 修复说明
12. 验证结果

### 7.7 状态流转

系统应根据当前状态和用户角色限制可执行动作。

示例规则：

| 当前状态 | 可执行动作 | 下一状态 |
|---|---|---|
| 待确认 | 确认为有效 | 已确认 |
| 待确认 | 驳回 | 已驳回 |
| 已确认 | 指派负责人 | 已分配 |
| 已分配 | 开始修复 | 修复中 |
| 修复中 | 提交验证 | 待验证 |
| 待验证 | 验证通过 | 已关闭 |
| 待验证 | 验证不通过 | 重新打开 |
| 重新打开 | 重新分派 | 已分配 |

每次状态变化必须记录：

1. 操作人
2. 操作时间
3. 原状态
4. 新状态
5. 操作说明

### 7.8 指派与处理

支持：

1. 指派负责人
2. 转派负责人
3. 设置优先级
4. 设置预计完成时间
5. 填写修复说明
6. 填写验证说明

### 7.9 评论协作

支持内部成员在 Bug 详情页评论。

评论能力包括：

1. 发表评论
2. 查看评论列表
3. 上传补充附件
4. 删除本人评论或管理员删除评论

第一版不做外部可见评论区，因为系统仅内部使用。

### 7.10 附件管理

支持上传和关联：

1. 图片截图
2. 已标注图片截图
3. 日志文件
4. 录屏文件
5. 其他辅助文件

建议限制：

1. 单个文件大小限制
2. 文件类型白名单
3. 附件归属 Bug
4. 上传人记录
5. 上传时间记录
6. 支持预览图片，下载日志和录屏
7. 支持对截图进行矩形框、箭头、线条、编号、文字等标注编辑


### 7.11 截图标注编辑器

Bug 提交和补充附件时，图片类附件需要支持在线标注编辑，方便提交人把问题位置说明清楚。

第一版建议支持：

| 能力 | 说明 |
|---|---|
| 矩形框标注 | 用方框圈出问题区域 |
| 箭头标注 | 指向具体问题位置 |
| 直线/自由线条 | 标记路径、边界或异常位置 |
| 编号标注 | 支持 `1`、`2`、`3` 等步骤或问题点编号 |
| 文字说明 | 在图片上输入简短说明 |
| 撤销/重做 | 防止误操作 |
| 删除选中标注 | 删除单个错误标注 |
| 颜色选择 | 至少支持红、黄、蓝等高对比颜色 |
| 线宽选择 | 支持不同粗细，适配截图尺寸 |
| 保存标注结果 | 将标注后的图片作为 Bug 附件提交 |
| 预览 | 提交前预览最终标注图 |

实现建议：

1. 标注编辑优先在前端浏览器本地完成。
2. 提交 Bug 时上传最终标注后的图片。
3. 建议同时保留原图和标注后图片，便于后续追溯。
4. 标注后的图片应在附件列表中明确展示为“已标注截图”。
5. 如果用户取消编辑，不应产生服务端附件记录。

### 7.12 统计看板

第一版建议保留基础统计：

1. 总 Bug 数
2. 待确认数量
3. 修复中数量
4. 待验证数量
5. 已关闭数量
6. 按项目统计
7. 按状态统计
8. 按严重程度统计
9. 按负责人统计

系统基于现有 RBAC 权限体系扩展菜单和按钮权限。

建议菜单：

1. Bug 管理
   - Bug 列表
   - 我的 Bug
   - Bug 提交
2. 项目管理
   - 老板驾驶舱
   - 项目概览
   - 需求管理
   - 迭代计划
   - 里程碑
   - 项目看板
   - 项目配置
   - 模块管理
   - 版本管理
   - Bug 看板

权限控制建议：

| 权限点 | 说明 |
|---|---|
| bug:create | 创建 Bug |
| bug:view | 查看 Bug |
| bug:update | 编辑 Bug 基础信息 |
| bug:assign | 指派负责人 |
| bug:status | 修改状态 |
| bug:comment | 评论 |
| bug:attachment | 上传附件 |
| bug:delete | 删除 Bug，建议仅管理员拥有 |
| bug:statistics | 查看统计 |
| project:manage | 管理项目、模块、版本 |

### 8.1 已确认角色搭建要求

第一版需要补齐以下角色和项目成员配置：

| 类型 | 必须项 | 可选项 |
|---|---|---|
| 系统角色 | `bug_project_owner`、`bug_product_owner`、`bug_reviewer`、`bug_developer`、`bug_tester`、`bug_submitter` | `bug_operator`、`bug_viewer` |
| 项目内角色 | `owner`、`product`、`reviewer`、`developer`、`tester` | `viewer` |
| 演示/验收用户 | `bug_owner`、`bug_product`、`bug_reviewer`、`bug_dev01`、`bug_dev02`、`bug_tester01`、`bug_submitter01` | `bug_viewer01`、`bug_operator01` |

权限边界要求：

1. `bug_reviewer` 必须能处理待确认池，执行确认、驳回、标记重复、暂不处理和指派。
2. `bug_developer` 只能看分派给自己的 Bug，不能看到未分派审核池。
3. `bug_viewer` 只能只读查看授权项目列表、详情和统计，不能提交、评论、上传或执行状态流转。
4. `bug_operator` 如果暂无运营/客服岗位，可以只保留角色，不创建实际用户。

## 9. 非功能需求

### 9.1 可用性

1. 表单字段清晰，降低提交成本。
2. 列表筛选高效，方便快速定位问题。
3. 状态流转按钮应根据权限和当前状态动态显示。

### 9.2 安全性

1. 所有接口必须鉴权。
2. 附件上传必须限制文件类型和大小。
3. 截图编辑器应在浏览器本地完成标注，避免原图未提交前产生无效服务端文件。
3. 用户只能访问自己权限范围内的数据。
4. 删除类操作需要权限控制和日志记录。

### 9.3 可追踪性

1. Bug 创建、编辑、分派、状态变更、评论、附件上传均需记录。
2. 操作记录应可在详情页查看。

### 9.4 可扩展性

1. 字典项尽量配置化，例如状态、类型、严重程度、优先级。
2. 状态流转规则建议集中维护，避免分散硬编码。
3. 项目、模块、版本作为独立基础数据维护。

第一版建议包含以下核心对象：

1. Bug

3. BugAttachment
4. BugHistory
5. Project

后续数据库设计阶段再细化字段、索引、关联和约束。

## 11. 验收标准草案

第一版完成后至少满足：

1. 内部用户可以登录后台并提交 Bug。
2. Bug 可以归属到不同项目、模块和版本。
3. Bug 可以按照确认的状态流程完整流转，且必须先经过 reviewer/产品/项目负责人审核后再分派开发。
4. 不同角色只能执行有权限的操作，developer 分派前不可见 tester 新提交 Bug。
5. Bug 详情页能查看描述、附件、评论和操作历史。
6. 支持上传截图、日志和录屏附件。
7. 上传 Bug 图片时可以打开图片编辑器，对截图画框、画线、箭头、编号和文字说明后再提交。
8. 管理员或项目负责人可以指派负责人。
9. 测试人员可以验证并关闭 Bug。
10. 列表支持常用筛选条件。
11. 基础统计看板可以展示项目内 Bug 概况。
12. 观察者只能只读查看被授权项目数据和统计，不能执行提交、评论、上传或状态流转。
13. 一期不出现 AI 修复入口，不调用本地 Codex，不创建 AI Runner。

## 12. 风险与注意事项

| 风险 | 说明 | 建议 |
|---|---|---|
| 状态流转过度硬编码 | 后续流程调整成本高 | 用状态流转配置集中管理 |
| 附件存储占用较大 | 录屏文件可能很大 | 设置大小限制，必要时接对象存储 |
| 权限边界不清 | 内部不同角色看到的数据范围不同 | 先明确角色矩阵，再实现接口 |
| 多项目数据隔离 | 项目之间数据可能需要隔离 | 查询接口必须考虑项目权限 |
| 列表筛选性能 | Bug 数量增多后查询变慢 | 关键字段建立索引 |

## 13. 待确认问题

进入下一阶段前，建议继续确认：

1. 是否需要“项目成员”概念，即每个项目绑定可见成员？
2. Bug 编号格式是否有要求，例如 `BUG-20260514-0001` 或按项目生成编号？
